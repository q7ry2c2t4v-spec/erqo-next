> Source: https://developer.apple.com/jp/design/human-interface-guidelines/context-menus

# コンテキストメニュー

コンテキストメニューを利用すれば、項目に関連する機能に直接アクセスできるので、インターフェイスを1つずつたどる手間が省けます。

![クリックしているポインタの下にある図案化されたコンテキストメニュー。画像は6色のオリジナルAppleロゴをイメージさせる赤色になっています。](https://docs-assets.developer.apple.com/published/a0d2b6a67b72d93b476903500cf282b7/components-context-menu-intro%402x.png)

コンテキストメニューはよく使う項目にすぐにアクセスできるので便利ですが、デフォルトでは非表示になっているので、ユーザが気付かない可能性があります。一般に、ユーザがコンテキストメニューを表示するには、現在の構成で利用できる入力モードを使ってビューまたはコンテンツを選択したあと、何らかのアクションを実行する必要があります。以下にアクションの例を示します:

  * visionOS、iOS、iPadOSで定義されたタッチまたはピンチして押さえたままにするジェスチャ

  * macOSおよびiPadOSで、Controlキーを押しながらポインティングデバイスでクリックする

  * macOSまたはiPadOSで、Magic Trackpadの副ボタンをクリックする

## [ベストプラクティス](/jp/design/human-interface-guidelines/context-menus#Best-practices)

**コンテキストメニューに含める項目を選ぶ際には関連性の高さを重視する。** コンテキストメニューの目的は、当該の状況でユーザが必要とする可能性が最も高いコマンドに素早くアクセスできるようにすることで、高度な機能やめったに使わない項目を提示することではありません。例えば、メールの受信トレイにあるメッセージのコンテキストメニューには、返信したりメッセージを移動したりするコマンドが含まれますが、メッセージの内容を編集したり、メールボックスを管理したり、メッセージをフィルタリングしたりするコマンドは含まれていません。

**メニュー項目の数を抑える。** コンテキストメニューが長すぎると、見渡したりスクロールしたりするのが難しくなります。

**アプリ全体で一貫してコンテキストメニューに対応する。** アプリ内の場所によってコンテキストメニューが表示されたり表示されなかったりすると、ユーザはどこでその機能を使えるのか予想できず、アプリに問題があると考えるかもしれません。

**コンテキストメニューの項目はメインのインターフェイス経由でも常に利用できるようにする。** 例えば、iOSとiPadOSのメールでは、受信トレイのメッセージで利用できるコンテキストメニューの項目は、メッセージビューのツールバーにも表示されます。macOSでは、アプリのメニューバーにあるメニュー経由で、さまざまなコンテキストメニューにあるコマンドを含めた、アプリのすべてのコマンドにアクセスできます。

**複雑なメニューを扱うためにサブメニューを使用する場合は1階層までにする。** サブメニューは、論理的に関係するコマンドを集めた子メニューが表示されるメニュー項目です。サブメニューを使うと、コマンドを整理してコンテキストメニューを短くできますが、サブメニューの階層を2つ以上にすると複雑で操作がしづらいとユーザが感じる可能性があります。サブメニューが必要な場合は、そのサブメニューを開かなくてもメニュー項目を推測できるような直感的なタイトルを付けてください。ガイダンスは、[サブメニュー](/jp/design/human-interface-guidelines/menus#Submenus)を参照してください。

**利用できないメニュー項目は淡色表示ではなく非表示にする。** 通常のメニューでは、利用できないアクションも表示しておくと、ユーザが実行できるアクションを見つけやすくなりますが、コンテキストメニューでは、現在選択されているビューやコンテンツに関連するアクションのみを表示するようにします。macOSでの例外は「カット」、「コピー」、「ペースト」のメニュー項目です。これらの項目は、現在のコンテキストに適用できない場合でも、利用できない状態で表示しておくことができます。

**最もよく使うメニュー項目をユーザの目にとまりやすい場所に配置する。** コンテキストメニューを開いたユーザは通常、その時点で自分の指やポインタがある場所に一番近い項目からメニューを読み始めます。多くのコンテキストメニューは、選択されたコンテンツがどこにあるかに応じて、その上か下に開きます。つまり、メニューの表示位置に合わせてメニュー内の項目の順序も逆にする必要が生じる可能性があります。

**キーボードショートカットはアプリのメインメニューに表示し、コンテキストメニューには表示しない。** コンテキストメニューは、それ自体がタスク固有のコマンドへのショートカットです。そこにキーボードショートカットも表示するのは冗長です。

**セパレータの使い方のベストプラクティスに従う。** その他のタイプのメニューと同様に、コンテキストメニューでもセパレータを使って項目をグループ化できます。これはユーザがメニューを素早く見渡すのに役立ちます。基本的に、1つのコンテキストメニュー内のグループの数は3つ以下にするのが賢明です。ガイダンスは[メニュー](/jp/design/human-interface-guidelines/menus)を参照してください。

**iOS、iPadOS、visionOSでは、データを破壊する可能性のあるコンテキストメニュー項目についてユーザに警告する。** コンテキストメニューに「削除」や「除去」などデータの破壊につながる可能性のある項目を含める必要がある場合は、それらをメニューの末尾に配置し、注意を促すための工夫をしてください（デベロッパ向けのガイダンスは、[`destructive`](/documentation/UIKit/UIMenuElement/Attributes/destructive)を参照してください）。破壊的なメニュー項目を赤のテキストで表示するようにすることもできます。

## [コンテンツ](/jp/design/human-interface-guidelines/context-menus#Content)

ほとんどのコンテキストメニューでタイトルは表示されません。これに対して、コンテキストメニューの項目には、その処理内容を明確に示す短いラベルを付ける必要があります。ガイダンスは、[メニュー＞ラベル](https://developer.apple.com/jp/design/human-interface-guidelines/menus#Labels)を参照してください。

**コンテキストメニューのタイトルは、それによってメニューの効果が明確になる場合にのみ表示する。** 例えば、メールでユーザがメッセージを複数選択し、iOSおよびiPadOSのツールバーの「マーク」ボタンをタップしたときに表示されるコンテキストメニューには、選択したメッセージの数を示すタイトルが表示されます。これは、次に選ぶコマンドが、選択されているすべてのメッセージに影響することをユーザに注意喚起するためです。

**メニュー項目のアクションを慣れ親しんだアイコンで表現する。** アイコンは、アプリ全体で共通するアクションをわかりやすく伝えるためのものです。「コピー」、「共有」、「削除」などのアクションは、表示される場所がどこであってもシステムと同じアイコンを使用して表現してください。一般的なアクションを表すアイコンのリストは、[標準のアイコン](/jp/design/human-interface-guidelines/icons#Standard-icons)を参照してください。追加のガイダンスは、[メニュー](/jp/design/human-interface-guidelines/menus)を参照してください。

## [プラットフォームの考慮事項](/jp/design/human-interface-guidelines/context-menus#Platform-considerations)

 _tvOSに追加の考慮事項はありません。watchOSには対応していません。_

### [iOS/iPadOS](/jp/design/human-interface-guidelines/context-menus#iOS-iPadOS)

**1つの項目に対してはコンテキストメニューか編集メニューのいずれかを表示し、両方は表示しない。** 同一の項目に対して両方のメニューを提示すると、ユーザが混乱するおそれがあります。システム側で操作の意図を識別するのも難しくなります。[編集メニュー](/jp/design/human-interface-guidelines/edit-menus)を参照してください。

**iPadOSのアプリ内でのオブジェクトの新規作成をコンテキストメニューから実行できるようにすることを検討する。** iPadOSでは、ユーザがタッチスクリーンを長押しするか、接続されたトラックパッドの副ボタンクリックや該当するキーボード操作を実行したときに、コンテキストメニューを表示できます。例えば「ファイル」では、ファイルやフォルダがない領域に表示したコンテキストメニューからフォルダを新規作成できます。

iOSおよびiPadOSのコンテキストメニューでは、コマンドリストの近くに現在のコンテンツのプレビューを表示できます。ユーザはメニューからコマンドを選択できるだけでなく、状況によっては、そのプレビューをタップして開いたり、別の領域までドラッグしたりできます。

**コンテキストメニューのコマンドの対象が明確になるよう、なるべくグラフィカルなプレビューにする。** 例えば、ユーザが「メモ」や「メール」のリストの項目上でコンテキストメニューを表示すると、プレビューに実際のコンテンツの縮小バージョンが表示され、意図通りの項目を操作しようとしていることを確認しやすくなります。

**プレビューのアニメーションの見た目がよくなるように配慮する。** ユーザが画面上のオブジェクトのコンテキストメニューを開くと、プレビューとメニューの背後にある画面が暗くなり、プレビュー画像がコンテンツから浮かび上がるようなアニメーションが表示されます。角丸などの輪郭がアニメーション中に変化しないように、プレビューのクリッピングパスをプレビュー画像の形状に合わせて細かく調整することは大切です。デベロッパ向けのガイダンスは、[`UIContextMenuInteractionDelegate`](/documentation/UIKit/UIContextMenuInteractionDelegate)を参照してください。

### [macOS](/jp/design/human-interface-guidelines/context-menus#macOS)

Macでは、コンテキストメニューは _コンテクスチュアル_ メニューと呼ばれることもあります。

### [visionOS](/jp/design/human-interface-guidelines/context-menus#visionOS)

**よく使う機能を提示する場合には、パネルやインスペクタウインドウではなくコンテキストメニューの使用を検討する。** アプリが開く個別のビューやウインドウの数を最低限に抑えると、ユーザが操作する画面をすっきりと保ちやすくなります。

**原則として、コンテキストメニューの高さがウインドウの高さを超えないようにする。** visionOSでは、ウインドウの上端よりも上および下端よりも下に、システムが提供するコンポーネント（ウインドウ管理コントロールや共有メニューなど）が表示されます。コンテキストメニューが縦に長すぎると、それらが隠れてしまうおそれがあります。ユーザがアプリをどのように使用する可能性が高いかを考え、それに基づいて、メニューに含める項目の数を決定してください。例えば、アプリを使用して高度な専門的タスクを実行するユーザは通常、多数の高機能コマンドの習得に時間を費やすので、コンテキストメニューからそれらのコマンドを利用できるようにしておくと喜ばれるかもしれません。一方、いくつかのシンプルなアクションしか実行しないユーザは、全体を素早く見渡せて使用できる短めのコンテキストメニューの方が嬉しいでしょう。
