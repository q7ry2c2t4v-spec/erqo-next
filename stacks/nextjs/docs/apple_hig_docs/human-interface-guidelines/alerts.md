# アラート

アラートを使って、ユーザが今すぐ必要とする重要な情報を提示できます。

![タイトル、説明文、プライマリボタン、セカンダリーボタンを含む図案化されたアラートのモックアップ。画像は全体が、6色のオリジナルAppleロゴを連想させる赤色になっています。](https://docs-assets.developer.apple.com/published/4502e6c9836f4d229ebc1fd2a549a0fc/components-alert-intro%402x.png)

例えば、ユーザに問題について伝えたり、アクションによってデータが破壊される可能性について警告したりできます。商品の購入やその他の重要なアクションの開始の確認をユーザに促す手段にもなります。

## [ベストプラクティス](/jp/design/human-interface-guidelines/alerts#Best-practices)

**アラートは控えめに使う。** アラートは重要な情報の提示に使えるとはいえ、そのたびにユーザの操作が中断されます。どのアラートにも必ず不可欠な情報や有用なアクションが含まれているようにしないと、ユーザがアラートを軽視するようになりかねません。

**情報提供の目的のみでアラートを使用しない。** ユーザは、アクションを必要としない、情報のみを提供するアラートによって中断されることを好みません。情報のみを提供する必要がある場合は、関連するコンテキスト内で情報を伝達する別の方法を検討してください。例えば「メール」では、サーバ接続が利用できない場合にインジケータが表示され、ユーザはそこから詳細を確認できます。

**破壊的なアクションでも、よく使う取り消し可能なアクションではアラートを表示しない。** 例えば、メールやファイルを削除するたびに、データが失われるアラートを表示する必要はありません。これは、ユーザが意図的にデータを破棄する操作で、そのアクションを取り消すこともできるからです。逆に、あまり使われない、破壊的でかつ元に戻せないアクションの場合は、意図せずにアクションが開始された可能性を考慮して、アラートを表示することが重要です。

**アプリの開始時にアラートを表示しない。** ユーザがアプリを開いたときに新しい情報や重要な情報について伝える必要がある場合は、その情報を確実に見つけられるほかの方法を工夫してください。ネットワーク接続が利用できないなどの問題がアプリの開始時に検出された場合も、それを知らせる別の方法を検討します。例えば、キャッシュされたデータやプレースホルダのデータを表示したり、ユーザの操作を中断しない形で問題について説明するラベルを表示することができます。

## [コンテンツ](/jp/design/human-interface-guidelines/alerts#Content)

すべてのプラットフォームで、アラートにはタイトル、オプションの情報テキスト、最大3つまでのボタンが表示されます。プラットフォームによって、アラートに追加の要素を含めることができます。

  * iOS、iPadOS、macOS、visionOSでは、アラートにテキストフィールドを追加できます。

  * macOSおよびvisionOSのアラートには、アイコンやアクセサリビューを追加できます。

  * macOSのアラートには、サプレッション[チェックボックス](/jp/design/human-interface-guidelines/toggles#Checkboxes)や[ヘルプボタン](/jp/design/human-interface-guidelines/buttons#Help-buttons)を追加できます。

**すべてのアラートの文言は、直接的、中立的で親しみやすい語調にする。** アラートは基本的に、問題や深刻な状況が説明する場所です。あいまいな表現や批判的な表現は避け、問題の深刻さが明確に伝わるようにします。

**簡潔かつ明確に状況を説明するタイトルにする。** ユーザに状況を素早く理解してもらう必要があるので、冗長な表現を避け、漏れなく具体的に記述します。できる限り、起きたこと、起きた状況、起きた理由について説明しましょう。「エラー」や「エラー329347が発生しました」のような、有用な情報を伝達しないタイトルにはしないでください。2行を超えて折り返されるような長すぎるタイトルも避けます。タイトルが完全な文である場合は、[センテンススタイルの大文字化（英語の場合）](https://help.apple.com/applestyleguide/#/apsgb744e4a3?sub=apdca93e113f1d64)を使用し、適切な文末記号を付けます。断片的な文をタイトルにする場合は、タイトルにふさわしい表現にし、文末記号は付けないようにします。

**情報テキストは有用な場合にのみ含める。** 情報メッセージを追加しなければならない場合は、センテンススタイルの大文字化を使用し（英語の場合）、適切な文末記号を付けた完全な文を使用して、できる限り短くします。

**アラートにボタンの説明を含めない。** アラートのテキストとボタンタイトルが明確であれば、何のためのボタンであるかを説明する必要はありません。ボタンの選択について説明が必要になる例外的なケースでは、それぞれのデバイスや操作方法の違いに配慮して、 _選択する_ などの表現を使用するようにし、ボタンタイトルは引用符をつけずに正確に参照してください。ガイダンスは、[ボタン](/jp/design/human-interface-guidelines/alerts#Buttons)を参照してください。

**対応するプラットフォームで、状況を解決するためにユーザの入力が必要な場合に限り、テキストフィールドを表示させる。** 例えば、パスワードを受け取るために安全なテキストフィールドを提示する必要がある場合があります。

## [ボタン](/jp/design/human-interface-guidelines/alerts#Buttons)

**ボタンのタイトルは簡潔かつ論理的にする。** ボタンのタイトルはできる限り、選択するとどうなるかを示す1語か2語のタイトルにしてください。アラートのテキストに直接関係する動詞や動詞句を使用するようにします。例えば、「すべてを表示」、「返信」、「無視」などです。情報アラートの場合は、「はい」や「いいえ」を使用せず、確認には「OK」を使用してください。アラートのアクションをキャンセルするボタンのタイトルは、必ず「キャンセル」にします。すべてのボタンのタイトルと同じように、[タイトルスタイルの大文字化](https://help.apple.com/applestyleguide/#/apsgb744e4a3?sub=apdca93e113f1d64)を使用し、文末記号は付けません。

**アラートが純粋に情報提供の目的でない限り、デフォルトのボタンのタイトルに「OK」を使用しない。** ユーザが行おうとしているアクションの確認を求めるアラートであっても、「OK」の意味が不明確になる場合があります。例えば、「OK、アクションを完了します」という意味になることも、「OK、アクションによって発生する否定的な結果について理解しました」という意味になることもあります。「消去」、「変換」、「クリア」、「削除」といった具体的なボタンのタイトルにすれば、実行されるアクションがユーザに理解しやすくなります。

**ボタンはユーザが予期する位置に配置する。** 通常、ユーザが最も選択する可能性が高いボタンは、横に並んだボタンの一番後ろ、または縦に並んだボタンの一番上に配置します。デフォルトのボタンは、必ず横に並んだボタンの一番後ろ、または縦に並んだボタンの一番上に配置してください。通常、キャンセルボタンは、横に並んだボタンの一番前、または縦に並んだボタンの一番下にあります。

**ユーザが意図して選択していない破壊的なアクションを実行するボタンには、破壊的であることが分かるようなスタイルを適用する。** 例えば、ユーザが意図して「ゴミ箱を空にする」などの破壊的なアクションを選択した場合、「ゴミ箱を空にする」ボタンはユーザが意図するアクションを実行するためのボタンです。そのため、表示されるアラートでは、「ゴミ箱を空にする」ボタンに破壊的なスタイルを適用しません。この例では、そのボタンが破壊的であることを再確認することよりも、意図して選択した「ゴミ箱を空にする」アクションをReturnキーを押して確定できる利便性が優先されます。逆に、ユーザが意図していない破壊的なアクションを実行できるボタンがある場合は、アラートで注意を促すことをおすすめします。

**破壊的なアクションがある場合は、キャンセルボタンを含めてそのアクションを安全に回避する明確な方法を提供する。** アラートのアクションをキャンセルするボタンのタイトルは、必ず「キャンセル」にします。キャンセルボタンをデフォルトのボタンにしないことが賢明です。ユーザがアラートをよく読むよう促し、自動的にReturnキーを押して閉じてしまわないようにしたい場合は、どのボタンもデフォルトにしないようにします。同様に、1つのボタンしかないアラートを表示する必要があり、それがデフォルトのボタンになる場合は、キャンセルボタンではなく完了ボタンを使います。

**適切な場合は、アラートをキャンセルする代替手段を用意する。** キャンセルボタンの選択のほかに、キーボードショートカットなどで画面のアラートを素早くキャンセルする方法があると便利です。以下に例を示します:

アクション| プラットフォーム  
---|---  
ホーム画面に戻る| iOS/iPadOS  
接続されたキーボードでEscape（Esc）キーまたはCommand＋ピリオド（.）キーを押す| iOS、iPadOS、macOS、visionOS  
リモコンのメニューを押す| tvOS  
  
## [プラットフォームの考慮事項](/jp/design/human-interface-guidelines/alerts#Platform-considerations)

 _tvOS、watchOSに追加の考慮事項はありません。_

### [iOS/iPadOS](/jp/design/human-interface-guidelines/alerts#iOS-iPadOS)

**ユーザが意図したアクションに関連する選択肢を提示する場合は、アラートではなくアクションシートを使用する。** 例えば、メールでメッセージの編集をキャンセルする場合、編集内容（または下書き全体）を削除、下書きを保存、編集に戻るという3つの選択肢がアクションシートで提示されます。アラートは、破壊的な結果につながるアクションの確認やキャンセルを行う場合に役立ちますが、アクションに関連する追加の選択肢を提示することはできません。ガイダンスは[アクションシート](/jp/design/human-interface-guidelines/action-sheets)を参照してください。

**スクロールが必要なアラートはできるだけ表示しない。** テキストのサイズが大きいとアラートのスクロールが必要になる可能性がありますが、アラートのタイトルを短くしたり、必要な場合のみ簡潔なメッセージを表示したりすれば、スクロールが必要になる可能性を最低限に抑えられます。

### [macOS](/jp/design/human-interface-guidelines/alerts#macOS)

macOSのアラートには自動的にアプリアイコンが表示されますが、代替のアイコンやシンボルを指定してもかまいません。さらに、macOSでは

  * ユーザが今後同じアラートの表示を抑制できるように、アラートの繰り返しを設定できます。

  * 必要に応じて、追加情報を提供するカスタムビューを付加できます（デベロッパ向けのガイダンスは、[`accessoryView`](/documentation/AppKit/NSAlert/accessoryView)を参照してください）。

  * ヘルプ文書を開くヘルプボタンを含めることができます（[ヘルプボタン](/jp/design/human-interface-guidelines/buttons#Help-buttons)を参照してください）。

**警告シンボルを多用しない。** アラートで「exclamationmark.triangle」のような警告シンボルを多用しすぎると、ユーザがアラートを軽視するようになりかねません。このようなシンボルは、予期しないデータの喪失につながる可能性があるアクションの確認など、実際に特別な注意が必要な場合に限って利用します。保存やゴミ箱を空にするなど、データの上書きや削除を行うだけのタスクでこのシンボルを使用しないでください。

### [visionOS](/jp/design/human-interface-guidelines/alerts#visionOS)

アプリが共有スペースで実行されているとき、visionOSのアラートはアプリのウインドウの前面、Z軸に沿って少し手前側に表示されます。

以下のカスタムコントロールを使用できます 

このビデオコンテンツの説明: visionOSの共有スペースで実行されているフリーボードアプリのアラートを示したビデオ。ビデオを再生すると、最近削除されたフリーボードのボードが完全に削除されます。その後、フリーボードウインドウの前面にアラートが表示され、確認が求められます。 

再生 

アラートを閉じずにウインドウを移動すると、アラートもウインドウと共に移動します。アプリがフルスペースで実行されている場合、アラートは装着者の[視野](/jp/design/human-interface-guidelines/spatial-layout#Field-of-view)の中央に表示されます。

以下のカスタムコントロールを使用できます 

このビデオコンテンツの説明: visionOSの共有スペースで実行されているフリーボードアプリのアラートを示したビデオ。ビデオを再生すると、最近削除されたフリーボードのボードが完全に削除されます。その後、フリーボードウインドウの前面にアラートが表示され、確認が求められます。アラートが消えることはなく、フリーボードウインドウに固定されたまま共有スペースを動き回ります。 

再生 

visionOSアラートにアクセサリビューを表示する必要がある場合は、高さ154 pt以下、コーナー半径16 pt以下のビューを作成してください。
