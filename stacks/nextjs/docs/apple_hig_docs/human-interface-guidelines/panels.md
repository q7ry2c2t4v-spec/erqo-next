# パネル

macOSアプリでパネルは通常、開いている別のウインドウの上に表示され、ユーザが位置を調整できます。パネルのコンテンツは、アクティブなウインドウまたは現在の選択内容に関連する補助的なコントロール、オプション、そのほかの情報です。

![ウインドウに重ねて表示されている図案化されたパネル。画像は全体が、6色のオリジナルAppleロゴを連想させる赤色になっています。](https://docs-assets.developer.apple.com/published/96a2ca8830165dd5be070df6d8edcbf9/components-panel-intro%402x.png)

一般に、パネルはアプリの[メインウインドウ](/jp/design/human-interface-guidelines/windows#macOS-window-states)より目立ちません。状況によっては、ヘッドアップディスプレイ（ _HUD_ ）体験に対応するために、パネルで黒っぽい半透明のスタイルが使われることがあります。

macOS以外のプラットフォームで動作するアプリで、現在のタスクや選択内容に関する補助的なコンテンツを表示したい場合は、モーダルビューの使用を検討してください。ガイダンスは[モダリティ](/jp/design/human-interface-guidelines/modality)を参照してください。

## [ベストプラクティス](/jp/design/human-interface-guidelines/panels#Best-practices)

**作業中のコンテンツに関する重要なコントロールや情報にユーザが素早くアクセスできるようにするために、パネルを使う。** 例えば、アクティブな書類やウインドウで選択した項目に影響するコントロールや設定は、パネルを使用して提供できます。

**インスペクタ機能の提示にはパネルを使う。** _インスペクタ_ は、現在選択している項目の詳細が表示されるコンポーネントで、項目が変更されたり、ユーザが新しい項目を選択したりすると、自動的にコンテンツが更新されます。一方、選択項目が変化しても常に同じコンテンツが維持される _情報_ ウインドウを提示する必要がある場合は、パネルではなく通常のウインドウを使用します。アプリのレイアウトによっては、[分割ビュー](/jp/design/human-interface-guidelines/split-views)パネルを使用してインスペクタを提示することも検討できます。

**パネルではなるべくシンプルな調整コントロールを使う。** テキスト入力や項目の選択では複数の手順が必要になり得ます。そのようなアクションを要求するコントロールはできる限り含めないでください。その代わりに、ユーザがもっと直接的に操作できるスライダやステッパーなどのコントロールを使用することをおすすめします。

**パネルの目的を説明する簡潔なタイトルを記述する。** 多くのパネルは、アプリ内で開いている別のウインドウに重ねて表示され、かつ移動が可能なので、ユーザがどこに置いても何のパネルか分かるようなタイトルバーが必要です。ユーザが何のパネルかを認識する助けになる、名詞または名詞句を使用した簡潔な[ふさわしい表現のタイトル](https://support.apple.com/guide/applestyleguide/c-apsgb744e4a3/web#apdca93e113f1d64)を作成してください。例えばmacOSでは、ユーザがすぐに分かる「フォント」や「カラー」などのパネルタイトルが使われています。「インスペクタ」というタイトルも多くのアプリでが使われています。

**パネルの表示/非表示を適切に切り替える。** アプリがアクティブになったときは、パネルが開いたときにどのウインドウがアクティブだったかを問わず、開いているパネルをすべて前面に移動します。アプリがアクティブでなくなったときは、すべてのパネルを非表示にします。

**パネルは「ウインドウ」メニューの書類リストに含めない。**[ウインドウメニュー](/jp/design/human-interface-guidelines/the-menu-bar#Window-menu)にパネルを表示/非表示にするコマンドを含めるのは問題ありませんが、パネル自体は書類や標準のアプリウインドウではないため、「ウインドウ」メニューのリストには含まれません。

**基本的に、パネルには最小化ボタンを含めない。** パネルは必要な場合のみ表示され、アプリがアクティブでなくなると表示されなくなるため、通常、最小化する必要はありません。

**インターフェイスとヘルプ文書でパネルに言及する際はタイトルを使用する。** メニューでは、 _パネル_ という用語は含めずにパネルのタイトルを使用します。例えば、「フォントを表示」、「カラーを表示」、「インスペクタを表示」などとします。ヘルプ文書に、異なるタイプのウインドウとして「パネル」という用語を導入すると、混乱の原因になりかねません。そのため通常は、パネルにはタイトルで言及するか、パネルタイトルに _ウインドウ_ を付加するとより明確になるのであれば、そうするのが最善です。例えば、「インスペクタ」というタイトルだけでコンテキストは十分に伝わります。一方、単に「フォント」や「カラー」とするよりも「フォントウインドウ」や「カラーウインドウ」とする方が明確になる場合もあります。

## [HUDスタイルのパネル](/jp/design/human-interface-guidelines/panels#HUD-style-panels)

HUDスタイルのパネルは、機能は標準のパネルと同じですが、見た目が黒っぽく半透明になります。HUDは、コンテンツのビジュアルを表示するアプリや、イマーシブな体験が売りのアプリに適しています。メディア編集やフルスクリーンのスライドショーなどがこれに当たります。例えばQuickTime Playerは、コンテンツの邪魔にならないように、HUDを使用してインスペクタ情報を表示しています。

![半透明のHUDパネル。ムービーコンテンツのファイル名、フォーマット、フレーム毎秒、データレート、フレームサイズなど、ムービーファイルのインスペクタ情報が表示されています。](https://docs-assets.developer.apple.com/published/dc8815ab50b3c4fc1c0ed072260b97d3/hud-style-panel%402x.png)

**標準パネルを優先する。** 妥当な理由なくHUDを使用すると、ユーザが戸惑ったり気が散ったりするおそれがあります。現在の外観の設定にHUDがなじまない場合もあります。通常は、次の場合に限ってHUDを使用してください:

  * ムービー、写真、スライドを表示するメディア指向アプリ

  * 標準のパネルでは重要なコンテンツが隠れる場合

  * コントロールを含める必要がない場合（開閉用三角ボタンを除き、システムが提供するほとんどのコントロールはHUDの見た目になじみません）

**アプリでモードを切り替えてもパネルのスタイルを維持する。** 例えば、アプリがフルスクリーンモードのときにHUDを使用する場合は、ユーザがフルスクリーンモードを解除しても、できる限りHUDスタイルを維持してください。

**HUDでは色の使用を控えめにする。** HUDの黒っぽい見た目で色を多用すると集中力が削がれる可能性があります。多くの場合、コントラストの高い色を少し使用するだけで、HUD内の重要な情報を強調表示できます。

**HUDは小さめにする。** HUDは、目立たないながらも有用な情報を提供できるように設計されています。HUDを大きくしすぎると、本末転倒になります。調整対象のコンテンツをHUDで隠さないでください。HUDが目立ちすぎてコンテンツからユーザの注意がそれないようにしてください。

デベロッパ向けのガイダンスは、[`hudWindow`](/documentation/AppKit/NSWindow/StyleMask-swift.struct/hudWindow)を参照してください。
