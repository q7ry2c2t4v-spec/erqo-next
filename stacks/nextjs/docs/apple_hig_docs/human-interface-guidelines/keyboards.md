# キーボード

物理キーボードは、テキストの入力、ゲームプレイ、アプリの制御などさまざまなタスクに使用される必須の入力デバイスと言えます。

![キーボードのスケッチ。キーボード入力を示しています。画像では長方形と円形のグリッド線が重なっており、6色のオリジナルAppleロゴをイメージさせる紫色になっています](https://docs-assets.developer.apple.com/published/c77cad58a6a9a5ca55d455583be012ca/inputs-keyboard-intro%402x.png)

物理キーボードは、Apple Watch以外のすべてのデバイスに接続できます。Macユーザは常に、iPadユーザはよく物理キーボードを使う傾向にあります。多くのゲームは物理キーボードとの相性がよく、テキストをたくさん入力する際には[仮想キーボード](/jp/design/human-interface-guidelines/virtual-keyboards)の代わりに物理キーボードを使うことが好まれます。

多くの場合、キーボードユーザは、キーボードショートカットを使ってアプリやゲームを素早く操作しています。 _キーボードショートカット_ は、1つのプライマリキーと1つ以上の修飾キー（Control、Option、Shift、Command）の組み合わせで、それが特定のコマンドに対応しています。ゲームのキーボードショートカット（ _キーバインド_ と呼ばれます）は、1つのキーからなる場合が多いです。

Appleは、システムとほとんどのアプリで一貫して動作する標準キーボードショートカットを定義しているので、知識を新たな体験に活用できます。一部のアプリでは、よく使われるアプリ固有のコマンドに対するカスタムのキーボードショートカットが定義されています。ほとんどのゲームでは、キーボードを使ってゲームを素早く効率的に操作できるカスタムのキーバインドが定義されています。ガイダンスは[ゲームコントロール](/jp/design/human-interface-guidelines/game-controls#Keyboards)を参照してください。

## [ベストプラクティス](/jp/design/human-interface-guidelines/keyboards#Best-practices)

**可能な限りフルキーボードアクセスに対応する。** iOS、iPadOS、macOS、visionOSで利用できるフルキーボードアクセスを使うと、ユーザはキーボードだけでウインドウ、メニュー、コントロール、システム機能をナビゲーションしたり、アクティベートしたりできます。アプリやゲームでフルキーボードアクセスをテストするには、システムが提供する設定アプリの「アクセシビリティ」領域で、フルキーボードアクセスをオンにします。デベロッパ向けのガイダンスは、[iOSアプリでフルキーボードアクセスに対応する](https://developer.apple.com/videos/play/wwdc2021/10120/)および[`isFullKeyboardAccessEnabled`](/documentation/AppKit/NSApplication/isFullKeyboardAccessEnabled)を参照してください。

重要

iPadOSは、テキストフィールド、テキストビュー、サイドバーでキーボードナビゲーションに対応しており、コレクションビューやカスタムビューで対応するために使用できるAPIを提供していますが、ボタン、セグメントコントロール、スイッチなどのコントロールをキーボードでナビゲーションできるようにはしないでください。代わりに、コントロールのアクティベート、すべてのオンスクリーンコンポーネントへのナビゲーション、ドラッグ&ドロップなどのジェスチャベースの操作には、フルキーボードアクセスを使うようにします。ガイダンスは[iPadOS](/jp/design/human-interface-guidelines/focus-and-selection#iPadOS)を参照してください。デベロッパ向けのガイダンスは[フォーカスベースの移動](https://developer.apple.com/documentation/uikit/focus-based_navigation)を参照してください。

**できる限り標準のキーボードショートカットを尊重する。** 通常のユーザは、ほとんどのアプリを使っているときに、ほかのアプリやシステム全体で動作する標準のキーボードショートカットが使えることを期待します。アプリでユーザが頻繁に実行する独自のアクションを提供する場合は、標準のショートカットを転用してユーザが別のアクションを関連付けるのではなく、[カスタム](/jp/design/human-interface-guidelines/keyboards#Custom-keyboard-shortcuts)のショートカットを作成するようにします。ユーザがゲームをプレイするときには、特定の標準的なキーボードショートカット（Command＋Qでゲームを終了するなど）が使用できるだけでなく、それぞれのゲームのキーバインドを個人のプレイスタイルに合うように変更することもできるはずだと考えています。ガイダンスは[ゲームコントロール](/jp/design/human-interface-guidelines/game-controls#Keyboards)を参照してください。

## [標準のキーボードショートカット](/jp/design/human-interface-guidelines/keyboards#Standard-keyboard-shortcuts)

**原則として標準のキーボードショートカットをカスタムアクションに使わない。** よく使うショートカットがアプリやゲームで異なる動きをすると、ユーザが混乱しかねません。標準のショートカットの再定義は、標準のアクションが体験では意味をなさない場合にのみ検討してください。例えば、テキスト編集作業に対応していないアプリの場合は、「イタリック」のような書式設定コマンドは必要ないので、Command＋Iキーを「情報を見る」などのより関連性の高いアクションに割り当てることができます。

## [カスタムキーボードショートカット](/jp/design/human-interface-guidelines/keyboards#Custom-keyboard-shortcuts)

**カスタムキーボードショートカットの定義はアプリ固有のコマンドの中で最もよく使うものに限定する。** 頻繁に実行するアクションにキーボードショートカットを割り当てておくとユーザに喜ばれますが、新しいショートカットをあまりにも多く定義しすぎると、習得するのが難しいアプリという印象を持たれてしまいます。

**ユーザが期待する方法で修飾キーを使用する。** Commandキーを押しながらドラッグすると複数の項目が1つのグループとして移動できたり、Shiftキーを押しながらドラッグでサイズ変更すると項目の縦横比を維持できたりするのはその例です。さらに、矢印キーを押している間、選択した項目をアプリで定義した最小距離単位で移動することもできます。

---|---|---

ヒント

言語によっては、修飾キーを使わないと入力できない文字があります。例えば、フランス語のキーボードではOption＋5で「{」の文字を入力します。通常は修飾キーにCommandを使っておけば安全ですが、どのキーボードでも利用できるわけではない文字と修飾キーの組み合わせは避ける方が賢明です。Command以外の修飾キーを使う必要がある場合は、できる限り英字とだけの組み合わせにしてください。

**修飾キーを正しい順序で表示する。** カスタムショートカットで複数の修飾キーを使う場合は、常に次の順序で表示してください: Control、Option、Shift、Command。

**キートップに文字が2つあるキーの上の方の字を使用するショートカットにShiftは追加しない。** キートップに文字が2つあるキーの上の方の字を入力するにはShiftキーを押したままにする必要があることはすでにユーザも理解しているため、ショートカットには単に上の方の字を表記するのが明快でしょう。例えば、「ステータスバーを非表示」のキーボードショートカットはCommand＋スラッシュ（/）となりますが、ヘルプのキーボードショートカットはCommand＋疑問符（?）で、Shift＋Command＋スラッシュ（/）ではありません。

**必要に応じてシステムによるキーボードショートカットのローカライズと表示を利用する。** 現在接続されているキーボードに合わせて、ショートカットのプライマリキーと修飾キーは自動的にローカライズされます。アプリやゲームで右から左のレイアウトに切り替える場合は、ショートカットが自動的に左右反転されます。ガイダンスは、[右から左](/jp/design/human-interface-guidelines/right-to-left)を参照してください。

**既存のショートカットに修飾キーを追加して、無関係なコマンドのショートカットを新規作成しない。** 例えば、Command＋Zはアクションを取り消す方法としてユーザに広く浸透しています。Shift＋Command＋Zを取り消し/やり直しと関係のないコマンドのショートカットとして使うと混乱の原因になりかねません。

### [visionOS](/jp/design/human-interface-guidelines/keyboards#visionOS)

visionOSでは、接続されたキーボードのCommandキーを押したままにすると開くショートカットインターフェイスに、アプリのキーボードショートカットが表示されます。Apple Vision Proのショートカットインターフェイスには、iPadまたはMacアプリの[メニューバーのメニュー](/jp/design/human-interface-guidelines/the-menu-bar)と同じような構造で、システム定義のメニューカテゴリ（ファイル、編集、表示など）にアプリコマンドが表示されます。ただし、メニューバーのメニューとは異なり、ショートカットインターフェイスではすべての関連カテゴリが1つのビューにまとめて表示され、それぞれのカテゴリには、ショートカットが設定されている利用可能なコマンドしか表示されません。

**ショートカットのタイトルは分かりやすくする。** ショートカットインターフェイスには各カテゴリ内のすべての項目がフラットなリストで表示され、その子項目の情報を示すサブメニュータイトルは表示されません。それぞれのショートカットのタイトルを分かりやすくして、サブメニューのタイトルが提供する追加コンテキストがなくても、アクションの内容が伝わるようにします。デベロッパ向けのガイダンスは、[`discoverabilityTitle`](/documentation/UIKit/UIKeyCommand/discoverabilityTitle)を参照してください。

**visionOSのアプリやゲームで物理キーボードを使う場合は、ユーザにオーバーレイ表示されることを認識する。** ユーザがvisionOSのアプリやゲームを使っているときに物理キーボードを接続すると、仮想キーボードのオーバーレイが表示され、タイプ入力の補完などのコントロールが提供されます。
