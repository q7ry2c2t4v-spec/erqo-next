# モーション

モーションが滑らかで美しければ、インターフェイスが生き生きとしたものになります。これにより、ステータスやフィードバック、指示を伝え、アプリやゲームの豊かな視覚体験を作ることができます。

![重なっている3つのひし形のスケッチ。左から右への要素の移動を示しています。画像では長方形と円形のグリッド線が重なっており、6色のオリジナルAppleロゴをイメージさせる黄色になっています](https://docs-assets.developer.apple.com/published/ff6bc56e55ddec4cbc4b129bbf81fc21/foundations-motion-intro%402x.png)

多くのシステムコンポーネントにはモーションが自動で含まれているため、アプリやゲーム全体でなじみのある一貫した体験を提供できます。システムコンポーネントは、アクセシビリティ設定やさまざまな入力方法のような要因に応じてモーションを調整することもあります。例えば、[Liquid Glass](/jp/design/human-interface-guidelines/materials#Liquid-Glass)の動きは、触覚による体験の感覚を強めるために一段と強調された直接的なタッチ操作に反応しますが、トラックパッドを使って操作したときに生まれる効果はより控え目です。

カスタムのモーションをデザインする場合は、以下のガイドに従ってください。

## [ベストプラクティス](/jp/design/human-interface-guidelines/motion#Best-practices)

**モーションの意図を明確にして体験を損なうことなく補助する。** モーションの追加自体を目的にしないでください。理由のないアニメーションや過剰なアニメーションはユーザの集中を妨げる原因になるほか、「まとまりがない」「身体的に不快」などと感じさせてしまうかもしれません。

**モーションはオプションにする。** 必ずしもすべてのユーザがアプリ内やゲーム内でモーションを体験できたり、体験したいわけではないため、重要な情報をアニメーションだけで伝えることは避ける必要があります。あらゆるユーザがアプリやゲームを楽しめるようにするため、[触覚](https://developer.apple.com/jp/design/human-interface-guidelines/playing-haptics)や[オーディオ](https://developer.apple.com/jp/design/human-interface-guidelines/playing-audio)などの別の方法でも伝えることで、視覚的フィードバックを補足してください。

## [フィードバックの提供](/jp/design/human-interface-guidelines/motion#Providing-feedback)

**ユーザのジェスチャと期待に沿うリアルなフィードバックモーションを目指す。** ゲーム以外のアプリの場合、リアルで正確なモーションはユーザが仕組みを理解する助けになりますが、意味のないフィードバックモーションは混乱のもとになることがあります。例えば、上部から下にスライドすることで表示されるビューの場合、まさか横にスライドしないと閉じることができないなどとは予測できません。

**フィードバックアニメーションは簡潔かつ正確に。** 簡潔さと正確性を併せ持つアニメーションフィードバックは軽快に感じられ、押しつけがましい印象を与えず、目立つアニメーションよりも効果的に情報を伝えられることが多いです。例えば、ゲームでアクションが成功したときに簡潔なアニメーションが正確なタイミングで表示されると、プレイヤーはゲームプレイから気が逸れることなく、すぐにアニメーションの意図を理解することができます。もう1つ、visionOSでの例を挙げます: 「写真」でパノラマをタップしたときには、パノラマが素早く滑らかに拡大してユーザ正面の空間を埋めつくします。これにより、ユーザがコンテンツを楽しむまでの待ち時間もなく、そのトランジションを追うことができます。

**アプリで頻繁に行われるUI操作には原則的にモーションを追加しない。** 標準のインターフェイス要素の操作にはシステムでわずかなアニメーションがすでに用意されています。カスタム要素の場合、通常はコンテンツが操作されるたびに不要なモーションを見せてユーザに余計な時間を使わせるということがないようにしてください。

**モーションをキャンセルできるようにする。** できる限り、アニメーションが完了するのを待たないとユーザは何も操作ができないという状況は避けてください。そのアニメーションをユーザが何度も体験する必要がある場合は特に注意してください。

**適切な場合はシンボルのアニメーションを使う。** SF Symbols 5以降では、SF Symbolsまたはカスタムシンボルにアニメーションを適用できます。ガイダンスは、[アニメーション](/jp/design/human-interface-guidelines/sf-symbols#Animations)を参照してください。

## [プラットフォームの機能を活用する](/jp/design/human-interface-guidelines/motion#Leveraging-platform-capabilities)

**対応する各プラットフォームでゲームのモーションがデフォルトで見栄えよく表示されるようにする。** ほとんどのゲームでは、フレームレートを常に30～60 fpsに維持することで、通常はスムーズで視覚的に魅力的な体験が得られます。対応するプラットフォームごとに、デバイスのグラフィックス機能を使って、ユーザが最初にその種の設定を変更しなくてもゲームを楽しめるようなデフォルト設定を有効にしてください。

**パフォーマンスやバッテリー駆動時間を最適化するためにユーザがゲームの視覚体験をカスタマイズできるようにする。** 例えば、システムが外部電源の存在を検知したときに、ユーザが電力モードを切り替えられるようにすることを検討してください。

## [プラットフォームの考慮事項](/jp/design/human-interface-guidelines/motion#Platform-considerations)

 _iOS、iPadOS、macOS、tvOSには追加の考慮事項はありません。_

### [visionOS](/jp/design/human-interface-guidelines/motion#visionOS)

visionOSでのモーションは、さりげなくコンテキストを伝えたり、情報を目立たせたり、イマーシブ体験を豊かなものにしたりするだけでなく、[奥行き](/jp/design/human-interface-guidelines/spatial-layout#Depth)と組み合わせて、ユーザがインタラクティブな要素を注視したときに重要なフィードバックを返すことができます。モーションはvisionOS体験の大部分を占める可能性があるため、ユーザの邪魔にならず、混乱や不快感を生じさせないようにすることが非常に重要です。

**ユーザの周辺視野でモーションを表示するのは可能な限り避ける。** 人間は、周辺視野での動きに特に敏感です。そうしたモーションは気が散るだけでなく、自分自身や自分の周囲が動いているように感じられるために不快感を生む原因にもなります。イマーシブ体験中に周辺視野で動いているオブジェクトを表示する必要がある場合は、オブジェクトの輝度レベルがほかの表示中のコンテンツと同等であることを確認してください。

**大きな仮想オブジェクトを動かして見せるときはユーザの快適性が保たれるように配慮する。** オブジェクトが[視野](/jp/design/human-interface-guidelines/spatial-layout#Field-of-view)の大半を埋めつくし、[パススルー](/jp/design/human-interface-guidelines/immersive-experiences#Immersion-and-passthrough)の大部分あるいは全体を覆い隠すほど大きい場合、人間はそれを周囲の物理環境の一部として知覚してしまう性質があります。ユーザが自分自身や自分の周囲が動いているかのように錯覚せずにオブジェクトの動きを知覚できるようにするには、オブジェクトの透明度を上げて向こう側が透けて見えるようにするか、オブジェクトのコントラストを下げてモーションを目立ちにくくします。

備考

ユーザは、ウインドウなどの大きな仮想オブジェクトを自分で動かしているときでさえも不快感を覚えることがあります。この場合も透明度やコントラストを調整することは有効ですが、ウインドウサイズをほどほどのサイズに保つことも検討してください。

**オブジェクトの位置を変える必要があるときはフェードの使用を検討する。** オブジェクトが1つの場所から別の場所に移動するとき、人間は自然とその動きを注視してしまいます。その動きがユーザにとって特段の意味を持たない場合は、移動の前にそのオブジェクトをフェードアウトし、新しい位置に移動したあとにフェードインしてください。

**一般にユーザが仮想世界を回転できるようにしない。** 仮想世界が回転すると、ユーザが回転を制御していて動きがわずかであっても、通常その体験はユーザの安定感を損ないます。そうした動きはさせずに、素早いフェードアウトの間に瞬時に方向を変えることを検討してください。

**固定された座標系を用意することを検討する。** 固定された領域の中で完結している視覚的な動きの方が、人間は楽に処理することができます。反対に、例えばプレイヤーに自動的に空間を移動させるゲームで、周囲のエリア全体が動いているように見えると、気分が悪くなってしまうことがあります。

**絶えず揺れ動くオブジェクトを見せることは避ける。** 人間が過敏に反応してしまう、振動数約0.2 Hzの振動を見せるのは特に避けることをおすすめします。揺れ動くオブジェクトを見せる必要がある場合は、揺れの大きさを控えめにすることを心がけてください。また、コンテンツを半透明にすることを検討してください。

### [watchOS](/jp/design/human-interface-guidelines/motion#watchOS)

SwiftUIは、アプリにモーションを追加するための強力で効率的な方法を備えています。WatchKitを使ってレイアウトや見た目の変更をアニメートする（またはアニメーションの画像シーケンスを作成する）場合は、[`WKInterfaceImage`](/documentation/WatchKit/WKInterfaceImage#1652345)を参照してください。

注意

レイアウトベースまたは外観ベースのすべてのアニメーションには、アニメーションの開始時と終了時に再生されるイージングが組み込まれます。イーシングをオフにしたりカスタマイズしたりすることはできません。
