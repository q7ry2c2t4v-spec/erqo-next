# 表現

アプリ内で使用する言葉の選択は、ユーザ体験にとって非常に重要です。

![書類と鉛筆のスケッチ。言葉が書かれたコンテンツを示しています。画像では長方形と円形のグリッド線が重なり、6色のオリジナルAppleロゴを反映させるための黄色になっています。](https://docs-assets.developer.apple.com/published/0fadb9fdee6277df9ea14009ca11a582/foundations-writing-intro%402x.png)

オンボーディング体験の構築、アラートの作成、アクセシビリティ向上のための画像についての説明文など、言語の視点に配慮しながらデザインすると、ユーザがアプリやゲームを最大限に活用しやすくなります。

## [はじめに](/jp/design/human-interface-guidelines/writing#Getting-started)

**アプリに適したボイスを決める。** 伝える相手が誰なのかを考慮すれば、使うべき用語や言葉づかいが明確になります。そのアプリのユーザにとって馴染みがあるのはどのような言葉でしょうか? ユーザにどのような印象を持ってもらいたいですか? 例えば、銀行アプリでは信頼性や安定感を伝え、一方、ゲームでは興奮や楽しい雰囲気を伝える言葉がよいでしょう。一般的な用語のリストを作成し、そのリストを参照して、使用する用語に一貫性を持たせるようにしましょう。用語に加え、アプリの価値を反映したボイスに統一することで、全体の一貫性が向上します。

**コンテキストに合ったトーンを使う。** アプリに適したボイスを確立したら、状況に応じてトーンを変えましょう。アプリの使用中にユーザが何をしているか、実世界での物理的な側面とアプリ内で起こっていることの両方の観点で検討してください。ユーザは今運動中で、目標を達成したところでしょうか。それとも、支払い時にエラーが発生してしまったのでしょうか。ユーザを取り巻く状況は、伝える内容そのものと、どのように画面にテキストを表示するかの両方に影響します。

次の2つのApple Watchの例のトーンを比べてみてください。1つ目は、簡潔かつ単刀直入なトーンで、状況の深刻さが表れています。2つ目は、明るく、祝福するようなトーンです。

![転倒検出メッセージのスクリーンショット。「ひどく転倒されたようです。」と表示されています。](https://docs-assets.developer.apple.com/published/d3ccbb1a4d1046a3e860ec9a46192e73/writing-fall-detection-message%402x.png)

![アクティビティのメッセージのスクリーンショット。「1日のムーブゴール達成の連続記録を更新しました: 35日です!」と表示されています。](https://docs-assets.developer.apple.com/published/d6b8d4ab180a754e9160f4a73c155ed2/writing-move-streak-message%402x.png)

**明確にする。** 理解しやすく、正しい情報が伝わるような言葉を選びましょう。それぞれの言葉を確認して、そこにその言葉が必要かどうかを見極めてください。もっと短い表現を使用できる場合は、そちらを使用します。迷った場合は、声に出して読み上げてみましょう。

**誰に対してもわかりやすい表現にする。** できるだけ多くのユーザに役立つアプリにするには、できるだけ多くのユーザに適した表現にする必要があります。簡潔で平易な言葉を選び、アクセシビリティとローカリゼーションを念頭に置いて、専門用語やジェンダー化された用語は避けましょう。ガイダンスは[インクルーシブな表現](https://help.apple.com/applestyleguide/#/apdcb2a65d68)および[VoiceOver](/jp/design/human-interface-guidelines/voiceover)と、デベロッパ向けのガイダンスは[ローカリゼーション](https://developer.apple.com/documentation/xcode/localization)を参照してください。

## [ベストプラクティス](/jp/design/human-interface-guidelines/writing#Best-practices)

**各画面の目的を考慮する。** 画面上の要素の順序に注意し、最も重要な情報を最初に配置します。読みやすくなるようにテキストをフォーマットしましょう。複数のアイデアを伝えたい場合は、テキストを複数の画面に分割することを検討し、画面間の情報の流れを検討してください。

**アクションに基づいた表現にする。** 能動的な文体と明確なラベルを使うことで、ユーザが手順に従ってアプリ内で次のステップに進んだり、画面間を移動したりしやすくなります。ボタンやリンクにラベルを付けるときは、ほとんどの場合、動詞を使用するのが最善です。分かりやすさを優先し、極端にくだけた表現や難解な表現を使用しないようにしましょう。例えば、多くの場合、「やりましょう!」とするより、単に「送信」とした方が効果的です。リンクでは、「ここをクリック」を使用するのは避け、「UX表現についての詳しい情報」など、内容が分かるような語句を使うようにします。この点は、特に画面読み上げ機能を使ってアプリにアクセスするユーザにとって重要です。

**文体や文言のパターンを構築する。** 一貫性を持たせると、ユーザにとって馴染みやすく、まとまりがあって直感的に操作でき、よく考えて設計されているアプリであるという印象を与えます。また、何度でもこれらのパターンに立ち返ることができるので、アプリの文言の作成が楽になります。

**アプリのスタイルに合った大文字化ルールを採用し、一貫して適用する。**[ボタンラベル](/jp/design/human-interface-guidelines/buttons#Content)など一部のコンポーネントには具体的なガイドラインがありますが、どのテキストフォーマットがアプリのボイスに合致するかを判断してください。タイトルケースは一般的にフォーマルな印象を与え、センテンスケースはより砕けた印象を与えます。UI要素のタイプごとにスタイルを選択し、アプリ内で一貫して使用してください。例えば、通知はすべてタイトルケース、見出しはすべてセンテンスケースのようにします。

**複数の手順があるプロセスでは、明確な指示を示し、用語や表現を統一する。** アプリに複数画面にまたがる操作手順がある場合は、次の手順に進むためのアクションのラベルの付け方を決めます。最初は「開始」のような表現を使って、手順を始めることを示します。ボタンラベルを使って次の手順を示すことも、「続ける」や「次へ」のような言葉を使うこともできますが、どれを選ぶかは統一してください。手順が完了したら、「完了」などの表現を使って明示します。

**所有代名詞は多用しない。** ほとんどの状況は、「 _私の_ 」や「 _あなたの_ 」などの所有代名詞がなくても説明できます。例えば、「あなたのお気に入り」よりも「お気に入り」の方が、より簡潔に同じ意味が伝わります。所有代名詞を使う場合は、アプリ全体で統一し、視点を切り替えないようにしてください。「 _私たち_ 」は誰を指しているのか曖昧になりがちなため、一切使わないでください。これは特に、「私たちはこのコンテンツを正常に読み込めません」のようなエラーメッセージで問題となります。「コンテンツを読み込めません」のようにする方が明確です。

**デバイスに応じた使用方法を記載する。** ユーザは、複数の種類のデバイスでアプリを使用する場合があります。使用する用語や表現はすべてのデバイスで統一する必要がありますが、さまざまなデバイスに適した表現になるように調整した方がよい部分について考えてください。ジェスチャは、デバイスに応じて適切に表現する必要があります。例えば、iPhoneやiPadなどのタッチデバイスで「タップ」を指す場合には、「クリック」という表現は使用しません。

ユーザがデバイスを使用する場所や方法、デバイスの画面サイズ、デバイスの場所は、すべてアプリでの表現に影響します。例えば、iPhoneとApple Watchでは、パーソナライズを可能にすることもできますが、画面が小さいため簡潔にする必要があります。一方、テレビの場合は、リビング空間にあることが多く、複数のユーザが画面を見る可能性があるので、誰を対象としているかを考慮してください。画面サイズが大きくても、離れた距離から見るユーザのためにテキストを大きくする必要があるので、ここでも簡潔さが求められます。

**空白の画面では明確な次の手順を示す。** 完了済みのTo Doリストや何も入っていないブックマークフォルダなどの空白の状態では、アプリについてもっと知ってもらう機会をユーザに提供できます。アプリやその特色などを紹介する場合は、コンテンツが役立つもので、コンテキストに合っていることを確認してください。空白の画面では次に何をするかが明確でないとユーザが躊躇してしまう可能性があるので、実行できる操作を提示したり、可能な場合は実行するためのボタンやリンクを表示しましょう。通常、空白の状態は一時的で、その画面はしばらくすると消えてしまう可能性があるため、重要な情報は表示しないでください。

**明確なエラーメッセージを表示する。** 常に、ユーザがエラーを事前に回避できるようにするのが最善策ですが、エラーメッセージを表示する必要がある場合は、できるだけ問題の近くに表示し、非難するような表現は避け、どうすれば解決できるかが分かるようにしてください。例えば、「パスワードが短すぎます」より、「8文字以上のパスワードを選択してください」の方が役立ちます。エラーは不満につながる可能性がある点に注意してください。「おっと!」や「あらまあ」などの間投詞は通常は不要で、不誠実に聞こえる可能性があります。言語表現だけでは対応できないエラーであり、多くのユーザに影響する可能性がある場合は、操作手順を考え直す機会と捉えてください。

**適切な表示方法を選択する。** そのアプリを積極的に使用しているかどうかに関わらず、ユーザの注意を惹きつける方法はたくさんあります。伝えたいことがある場合は、メッセージの緊急度と重要度を検討します。メッセージが表示される状況、すぐにアクションを起こしてもらう必要があるかどうか、および必要とされるサポート情報の量を検討してください。適切な表示方法を選択し、状況に合ったトーンを使用しましょう。ガイダンスは、[通知](/jp/design/human-interface-guidelines/notifications)、[アラート](/jp/design/human-interface-guidelines/alerts)、[アクションシート](/jp/design/human-interface-guidelines/action-sheets)を参照してください。

**設定のラベルは明確かつシンプルにする。** できるだけ実用的なラベルを付けて、設定を見つけやすくしてください。設定のラベルだけでは分かりにくい場合は、説明を追加します。オンにすると何が起きるかを説明し、オンにしなかった場合の状況をユーザが推測できるようにします。例えば、Apple Watchの手洗いタイマーでは、手を洗っているときにタイマーを開始できるという説明が表示されています。この設定がオフになっているとタイマーは開始されないことを伝える必要はありません。

![手洗いタイマーの説明が表示されているスクリーンショットの一部。次のように表示されています: 「Apple Watchで、手洗いを検出して20秒のタイマーを開始することができます。」](https://docs-assets.developer.apple.com/published/cf02bca505593df48203f0711e4aa7d0/writing-handwashing-settings%402x.png)

設定画面に誘導する必要がある場合は、その画面へのアクセス場所を記載するのではなく、直接的なリンクやボタンを提供します。ガイダンスは、[設定](/jp/design/human-interface-guidelines/settings)を参照してください。

**テキストフィールドにはヒントを表示する。** アプリでユーザにアカウントや連絡先などの情報を入力してもらう場合は、すべてのフィールドに分かりやすいラベルを付け、ヒントやプレースホルダを使用して情報の入力形式が分かるようにします。「name@example.com」などのヒントテキストで例を示したり、「自宅」など、入力してもらう情報について説明します。エラーはフィールドのすぐ横に表示し、ルールに従っていないことを責めるのではなく、正しく情報を入力する方法を伝えます。「数字や記号を使用しないでください」より、「名前には文字のみを使用してください」とした方が効果的です。「無効な名前」など、役立つ情報が含まれない機械的なメッセージは避けてください。ガイダンスは、[テキストフィールド](/jp/design/human-interface-guidelines/text-fields)を参照してください。
