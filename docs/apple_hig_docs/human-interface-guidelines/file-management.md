# ファイルの管理

一部のアプリでは、ユーザがシステム全体で管理できると期待している書類やファイルに対応できます。

![右上隅が折りたたまれた書類のスケッチ。ファイルの操作を示しています。画像では長方形と円形のグリッド線が重なっており、6色のオリジナルAppleロゴをイメージさせるオレンジ色になっています。](https://docs-assets.developer.apple.com/published/7b051f472625f03af6cf5c5e83b1ed58/patterns-file-management-intro%402x.png)

Pages、Keynote、写真、プレビューなどドキュメントベースアプリでは、書類やファイルを簡単に作成、編集、保存することができ、ほとんどの場合、アプリで開きたいコンテンツを独自の方法でブラウズできるようになっています。

また、ユーザはドキュメントベースアプリを開かなくても書類をブラウズできると期待しています。例えばMacでは、Finderを使ってmacOSファイルシステムにアクセスします。iPhone、iPad、およびApple Vision Proでは、ファイルアプリを使ってデバイス上の書類やファイルを管理します。watchOSおよびtvOSでは、通常、書類の作成、編集、管理は行わないため、これらのシステムでは書類ブラウズのインターフェイスは提供されません。

## [ファイルを作成する/開く](/jp/design/human-interface-guidelines/file-management#Creating-and-opening-files)

**アプリのメニューとキーボードショートカットを使って、書類を作成して開く便利な方法を提供する。** iPadOSおよびmacOSのユーザは、慣れ親しんだメニューコマンドを使って、書類を新規作成したり、既存の書類を開いたりできることを期待します。「新規」や「開く」などのコマンドを提供すると、iPadOSでは、接続されたハードウェアキーボードのCommandキーを押したままにすると開くショートカットインターフェイスに表示されます。macOSでは、「ファイル」メニューのメニューバーに表示されます。キーボードショートカットが利用可能かどうかにかかわらず、新規書類を作成できる「追加」（+）ボタンを含めてください。macOSアプリでは、追加アクションは「ファイル」メニューに配置します（ガイダンスは、[「ファイル」メニュー](/jp/design/human-interface-guidelines/the-menu-bar#File-menu)を参照してください）。

**アプリにカスタムのファイルブラウザが必要な場合は、ユーザがそのプラットフォームのファイルシステムを理解できるよう手助けする。** Finderアプリとファイルアプリに慣れ親しんでいるユーザは、デバイスのファイルシステムの基本的なレイアウトをすでに理解しています。カスタムのファイルブラウザが開いたときには、ファイルシステムの最も関連する部分（書類フォルダやiCloudフォルダ、最近選択した場所など）を表示することをおすすめしますが、ファイルシステムのほかの部分もそのブラウザで表示できるようにしてください（必要な場合）。

## [作業内容の保存](/jp/design/human-interface-guidelines/file-management#Saving-work)

**キャンセルしたり削除したりしない限り、作業内容は常に保存されているとユーザが確信できるようにする。** 基本的には、ユーザが作業内容を明示的に保存する必要がないようにします。その代わり、ユーザの編集作業中に定期的に自動保存を行い、ユーザがファイルを閉じたり別のアプリに切り替えたりするタイミングでも保存を実行してください。

**ファイル拡張子はデフォルトでは隠し、必要に応じてユーザが表示できるようにする。** 保存やファイルを開くすべてのインターフェイスに現在の選択を確実に適用してください。

## [Quick Lookのプレビュー](/jp/design/human-interface-guidelines/file-management#Quick-Look-previews)

Quick Lookを利用すると、アプリで取り扱うファイルのプレビューを簡単に生成できます。ユーザは、それらをアプリ内で表示したり、場合によっては操作したりできます。例えば、Quick Lookを使用して、オーディオファイルのプレビューを聴いたり、写真のプレビューにマークアップを追加したり、3Dファイルのプレビューを回転または拡大/縮小してさまざまな方法で見たりできます。

**Quick Lookビューアを使用して、そのアプリでは開けないファイルでもプレビューできるようにする。** アプリが対応していないファイルについて添付などの操作を可能にする場合、Quick Lookビューアを実装すれば、それらのファイルのプレビューをアプリ内で提供できます。

**アプリで出力されるファイルがカスタムのファイルタイプである場合は、Quick Lookジェネレータの実装を検討する。** Quick Lookジェネレータを利用すれば、アプリで出力した書類のプレビューをFinder、ファイル、Spotlightなどのほかのアプリに表示できるので、ユーザから見つけやすくなります。

### [iOS/iPadOS](/jp/design/human-interface-guidelines/file-management#iOS-iPadOS)

#### [書類ランチャー](/jp/design/human-interface-guidelines/file-management#Document-launcher)

iOS 18およびiPadOS 18以降では、ドキュメントベースアプリでシステムの書類ランチャーを使用することで、一貫した非常にグラフィカルな方法でファイルをブラウズしたり、開いたり、作成したりできます。書類ランチャーではアプリのテーマの主要な要素が強調されるフルスクリーン体験が提供され、新規書類を作成しやすくなります。デベロッパ向けのガイダンスは、[`DocumentGroupLaunchScene`](/documentation/SwiftUI/DocumentGroupLaunchScene)を参照してください。

書類ランチャーは以下の3つの主要な部分から構成されます:

  * アプリのタイトルと、アプリ固有の2つのボタンが表示された _タイトルカード_

  * タイトルカードの後ろに表示される背景画像と、その周囲に表示できる追加の画像（ _アクセサリ_ と呼ばれます）

  * ファイルブラウザおよび（オプションとして）アプリ固有のコントロールが含まれるシート

書類ランチャーの3つの部分はすべてカスタマイズすることができます。タイトルカードにはアプリ名が自動的に表示されますが、カードの主ボタンと副ボタンのテキストと機能は指定する必要があります。また、カスタムの背景画像や、タイトルカードの周囲に表示されるアクセサリ画像を1つ以上作成したり、ファイルブラウザのツールバーに表示できるカスタムコントロールを提供することもできます。

![横向きのiPadの作文アプリの書類ランチャーのスクリーンショット。書類ランチャーには、カスタムの背景と2つのアクセサリ画像が表示されています。下部には、ファイルブラウザのシートに次の3つのタブが表示されています: 「最近の項目」、「共有」、「ブラウズ」。](https://docs-assets.developer.apple.com/published/7973e6412ab72fac5ea035089a1abca7/file-management-document-launcher%402x.png)

**タイトルカードのボタンはアプリの最も重要な機能に割り当てる。** 通常、主ボタンでは新規書類が作成され、副ボタンではその他のオプションを提供できます。例えば、Numbersの主ボタンは「書き始める」、副ボタンは「テンプレートを選択」です。

**アクセサリやタイトルカードとは明確に異なる背景にする。** 単色、グラデーション、またはパターンを使用できます。前面の要素から気を散らすおそれのある複雑な画像やパターンは含めないでください。

**アクセサリの配置に配慮する。** 例えば、アクセサリをタイトルカードの前面と背面の両方に配置して奥行きを表現することができますが、アプリ名と両方のボタンがはっきり見えるようにしてください。タイトルカードをあまりに多くのアクセサリで散らかすのは避け、対応している範囲の画面サイズとデバイスの向きで、全体の見た目をテストしてください。

**アニメーションは控えめにする。** ディスプレイに動きがありすぎると、ユーザの混乱のもとになります。アクセサリをアニメートさせたい場合は、アプリのコンテンツをさりげなく強調して高めるような、控えめに繰り返すアニメーションを作成することを検討してください。例えば、アクセサリがそっと呼吸したり揺れたりするように見えるアニメーションを作成することをおすすめします。ガイダンスは[モーション](/jp/design/human-interface-guidelines/motion)を参照してください。

#### [ファイルプロバイダアプリ機能拡張](/jp/design/human-interface-guidelines/file-management#File-provider-app-extension)

アプリのファイルをほかのアプリとの間で共有する機能がある場合は、ファイルプロバイダアプリ機能拡張を作成して、アプリの書類を読み込み/書き出し、開き、移動するカスタムインターフェイスを表示できます。デベロッパ向けのガイダンスは、[File Provider](/documentation/FileProvider)を参照してください。 _アプリ機能拡張_ は、デベロッパからユーザに提供されるインストール可能なコードで、システムの特定の領域の機能を拡張します。詳しくは、[アプリ機能拡張](https://developer.apple.com/app-extensions/)を参照してください。

**提供したファイルプロバイダ機能拡張を使ってユーザが書類を開いたり読み込んだりしようとする場合は、現在の作業に適切な書類のみを表示する。** 例えば、PDF編集アプリが書類を読み込もうとしている場合は、開くか読み込む対象となるPDFファイルのみのリストを表示します。必要に応じて、変更日、サイズ、書類の場所がローカルかリモートかなど、追加の情報を表示してもかまいません。

**書類を書き出したり移動したりする場合は、ユーザが出力先を選択できるようにする。** アプリの書類の保存先が単一のディレクトリに限定されるのでない限り、アプリのディレクトリ階層内をユーザが移動できるようにします。新しいサブディレクトリを追加する方法を提供してもよいでしょう。

**カスタムの上部のツールバーはできるだけ使わない。** 機能拡張が読み込まれるモーダルビューにはすでにツールバーがあります。ツールバーが2つもあると混乱の元となり、コンテンツを表示するスペースも減ります。

ユーザがほかのアプリからアプリのファイルをブラウズしたり開いたりできるようにも設計できます。デベロッパ向けのガイダンスは、[アプリに書類ブラウザを追加する](/documentation/UIKit/adding-a-document-browser-to-your-app)を参照してください。

### [macOS](/jp/design/human-interface-guidelines/file-management#macOS)

#### [カスタムファイル管理](/jp/design/human-interface-guidelines/file-management#Custom-file-management)

ユーザは、Finderやほとんどのドキュメントベースアプリで慣れ親しんだファイルブラウズ体験に強い親近感を覚えます。カスタムのファイルブラウザを作成する重要な理由がない限りは、デフォルトのファイルブラウザを使用してください。

**ファイルを開くカスタムインターフェイスの利便性を確保する。** 例えば、単純な「開く」操作に加えて「最近使った項目を開く」操作があると、ユーザにとって便利かもしれません。ユーザがファイルブラウズ時にフィルタリング条件を指定したり、複数の書類を選択して同時に開いたりできるようにするのも、利便性を高めるアイデアの1つです。macOSのファイルを開くパネルでは、タスクに合わせて「開く」ボタンのタイトルをカスタマイズできます。例えば、アプリでファイルの内容を現在の書類に挿入できる場合は、タイトルを「挿入」に変更できます。

**保存するファイルの名前、形式、場所をユーザが変更できるインターフェイスを提供する。** デフォルトでは、新しい書類のタイトルは「名称未設定」となります。ユーザはあとで別の名前に変更できます。書類を開くインターフェイスの場合と同様、保存ビューのブラウズ操作でも、ユーザにとって利便性が高いと考えられる場所を書類保存のデフォルトとして提示できます。コンテンツをさまざまな形式で保存できるアプリでは、ユーザがファイル形式を指定できる方法も提供してください。

**「保存」ダイアログの機能を拡張することを検討する。** 必要であれば、「保存」ダイアログに、便利な設定やオプションを含むカスタムアクセサリビューを追加できます。例えば、「メール」のメッセージをファイルとして保存するダイアログには、添付ファイルを含めるオプションが表示されます。

#### [Finder Sync機能拡張](/jp/design/human-interface-guidelines/file-management#Finder-Sync-extensions)

アプリでローカルとリモートのファイルを同期する場合、Finder Syncアプリ機能拡張を作成して、Finder内にファイルの同期状況とコントロールを表示できます。デベロッパ向けのガイダンスは、[Finder Sync](/documentation/FinderSync)を参照してください。

例えば、Finder Sync機能拡張を以下のように利用できます:

  * 複数項目の同期状況を示すバッジをFinderに表示する

  * お気に入りに追加する、パスワードで保護するなど、ファイルおよびフォルダの管理タスクを実行するカスタムのコンテキストメニュー項目を作成する

  * 同期処理を開始するなどのグローバルなアクションを実行するカスタムのツールバーボタンを作成する

**自動保存をオフにしてもユーザが作業内容を失わないようにする。** ユーザが「デスクトップとDock」設定で「書類を閉じるときに変更内容を保持するかどうかを確認」トグルを選択して、自動保存をオフにする場合があります。このようなシナリオでは、ユーザが書類を閉じたり、アプリを終了したり、ログアウトしたり、再起動したりしようとするときに、保存されていない変更内容が書類に含まれていることを通知して、「保存」ダイアログを開いてください。

**自動保存がオフになっている場合で、保存されていない変更内容が書類に含まれているときは、必ずユーザに分かるようにしておく。** 保存されていない変更内容があることを示すため、書類ウインドウの閉じるボタン、およびアプリの「ウインドウ」メニューの書類の名前の横にドットを表示してください。ドットは作業内容を保持するためにユーザの操作が必要であることを示すものなので、自動保存がオンになっている場合にこれらの場所にドットを表示すると混乱します。自動保存のオン/オフにかかわらず、タイトルバーの書類のタイトルの末尾に「編集済み」と表示できますが、自動保存が起動したか、ユーザが明示的に作業内容を保存したらすぐにこの文字列を消去してください。
