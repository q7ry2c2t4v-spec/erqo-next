# オンボーディング

オンボーディングによって、アプリやゲームをすぐに使い始めることができます。

![手を振っているスケッチ。歓迎のジェスチャを示しています。画像では長方形と円形のグリッド線が重なっており、6色のオリジナルAppleロゴをイメージさせるオレンジ色になっています。](https://docs-assets.developer.apple.com/published/f35d65fafbb859b1d7578300dff0f12c/patterns-onboarding-intro%402x.png)

アプリやゲームはただ使用するだけで理解できるのが理想的ですが、オンボーディングが必要な場合は、短時間で楽しく終えることができ、スキップ可能なフローをデザインするようにしましょう。オンボーディングを利用できるようにする場合は、[起動](/jp/design/human-interface-guidelines/launching)の完了後に行います。オンボーディングは起動体験の一部ではありません。

## [ベストプラクティス](/jp/design/human-interface-guidelines/onboarding#Best-practices)

**インタラクティブな方法で説明する。** 説明をただ読むだけではなく、実際に学んだタスクを行ってみる方が、情報を理解して覚えやすくなります。できる限りインタラクティブなオンボーディング体験を提供し、安全にアクションを試したり、機能を発見したり、ゲームの仕組みを試したりできるようにします。

**1つのオンボーディングフローではなく、コンテキストに固有なヒントを集めて提供することを検討する。** 同じようなコンテキストで利用できるヒントをまとめて体験に組み込むと、アプリやゲームを進めながら、現在のタスクについて学ぶことができます。コンテキストに固有なヒントは、学習効果の向上にも役立ちます。新しい情報に遭遇する前に、1つのアクションやタスクに集中できるからです。インターフェイスの特定の領域について説明するコンテンツがある場合は、その領域の近くに説明を表示します。デベロッパ向けのガイダンスは、[TipKit](/documentation/TipKit)を参照してください。

**前提条件となるオンボーディングフローを表示する必要がある場合は、覚える情報を詰め込みすぎず、簡潔で楽しい体験をデザインする。** オンボーディングが短く楽しいと、ユーザに完了してもらえる可能性が高くなります。逆に、たくさんのことを教えようとしすぎると、ユーザは圧迫を感じ、学んだことを覚える可能性が低くなる場合があります。

**適宜、チュートリアルを分割することや省略可能にすることを検討する。** アプリやゲームを最初に起動するときにチュートリアルをスキップできるようにする場合は、次回以降に起動したときに再表示しないようにします。ただし、あとから見たくなった場合のために、簡単に見つけられるようにしてください。例えば、アプリやゲームのヘルプ、アカウント、設定といった領域からチュートリアルを利用できるようにします。

**オンボーディングコンテンツではアプリやゲームで提供される体験に集中する。** ユーザがオンボーディングフローに入るのは、アプリやゲームについて知るためです。システムやデバイスの使い方を知る必要はありません。

## [追加コンテンツ](/jp/design/human-interface-guidelines/onboarding#Additional-content)

**必要な場合は短いスプラッシュスクリーンを表示する。** スプラッシュスクリーンを含める必要がある場合は、簡潔に伝わる美しいグラフィックスをデザインします。スプラッシュスクリーンは、体験が遅くなると感じさせることのないように、一目で情報を吸収するために十分な長さだけ表示するようにしてください。

**大量のダウンロードでオンボーディングを妨げない。** ユーザは、オンボーディングフローを行うかどうかにかかわらず、初めて起動したあと、アプリやゲームをすぐに使い始めたいと思っています。ソフトウェアパッケージにメディアなどのコンテンツを十分に含め、ユーザがダウンロードを待たずにアプリやゲームを操作できるようにすることを検討してください。ガイダンスは、[起動](/jp/design/human-interface-guidelines/launching)を参照してください。

**オンボーディングのフローでライセンスの詳細情報を表示しない。** 使用許諾や免責事項はApp Storeに表示し、ユーザがアプリやゲームをダウンロードする前に読むことができるようにしてください。これらの項目をオンボーディングフロー内に含める必要がある場合は、体験を損なわないようにバランスのとれた方法で組み込むようにしましょう。

## [追加リクエスト](/jp/design/human-interface-guidelines/onboarding#Additional-requests)

**必須でない設定フローやカスタマイズ手順はあとで行う。** ほとんどのユーザが追加設定を行うことなく、すぐにアプリやゲームを使い始めることができるように、妥当なデフォルト設定を提供します。

**アプリやゲームの動作にプライベートなデータやリソースへのアクセスが必要な場合は、許可のリクエストをオンボーディングフローに組み込むことを検討する。** このシナリオでは、リクエストをオンボーディングのフローで表示すると、アプリやゲームでユーザの許可が必要になる理由と、それを許可するメリットについて説明できる機会になります。そうしない場合は、プライベートなデータやリソースを使用する特定の機能に初めてアクセスしたときに、許可リクエストを表示します。ガイダンスは、[許可のリクエスト](/jp/design/human-interface-guidelines/privacy#Requesting-permission)を参照してください。

**評価や購入を促す前にアプリやゲームを利用できるようにする。** アプリやゲームを本格的に利用してからの方が、そのようなリクエストに対してより前向きな反応を示してもらえます。

## [プラットフォームの考慮事項](/jp/design/human-interface-guidelines/onboarding#Platform-considerations)

 _iOS、iPadOS、macOS、tvOS、visionOS、watchOSに追加の考慮事項はありません。_
