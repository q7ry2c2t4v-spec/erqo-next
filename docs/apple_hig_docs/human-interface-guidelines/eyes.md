# 視線

visionOSでは、仮想オブジェクトを注視することで、操作の対象として判別します。

![人間の目のスケッチ。画像では長方形と円形のグリッド線が重なっており、6色のオリジナルAppleロゴをイメージさせる紫色になっています](https://docs-assets.developer.apple.com/published/126393ded1c486236fc7a9feabea30ea/inputs-eyes-intro%402x.png)

インタラクティブな要素を注視すると、それが強調表示されることによって視覚的なフィードバックが返されます。ユーザはこのフィードバックをもとに、それが目的の項目かどうかを確かめます。この視覚的なフィードバック（ _ホバーエフェクト_ ）は、タップなどの[間接的ジェスチャ](/jp/design/human-interface-guidelines/gestures#visionOS)を使って要素を操作できることを示します。

以下のカスタムコントロールを使用できます 

このビデオコンテンツの説明: 設定アプリの録画。視線がある設定から別の設定に移動するにつれて、順番にホバーエフェクトが表示されます。 

再生 

場合によっては、注視したコンポーネントが自動的に拡大されます。例えば、タブバーを注視すると、タブバー全体のサイズが変わって各タブの横にテキストラベルが表示されます。この場合は、タブバーが拡大する前に対象のタブが強調表示されるので、ラベルが表示される前にそのタブを選択することもできます。また、注視したときにツールチップを表示できるボタンもあります。

重要

ユーザのプライバシーを保護するため、ユーザがタップの前に注視している場所に関する直接的な情報を、visionOSが提供することはありません。システムが提供するコンポーネントを使用している場合、ユーザがそのコンポーネントをタップしたことはvisionOSからアプリに自動的に通知されます。デベロッパ向けのガイダンスは、[プライバシーおよびユーザ設定のベストプラクティスを取り入れる](/documentation/visionOS/adopting-best-practices-for-privacy)を参照してください。

visionOSは、 _フォーカスエフェクト_ にも対応しています。これは、キーボードやゲームコントローラなどの接続された入力デバイスを使ってアプリやシステムを操作する際に役立ちます。フォーカスエフェクトはホバーエフェクトとは関係がありません。詳しくは、[フォーカスと選択](/jp/design/human-interface-guidelines/focus-and-selection)を参照してください。

## [ベストプラクティス](/jp/design/human-interface-guidelines/eyes#Best-practices)

**アプリを操作する方法を必ず複数用意する。** アプリをデザインする際には、ユーザがデバイスの操作方法をパーソナライズできるようにアクセシビリティ機能に対応させてください。ガイダンスは[アクセシビリティ](/jp/design/human-interface-guidelines/accessibility)を参照してください。

**視覚的に快適に使えるデザインにする。** ユーザが主要なタスクを実行しやすくなるように、使う必要のあるオブジェクトが確実に[視野](/jp/design/human-interface-guidelines/spatial-layout#Field-of-view)の中に表示されるようにしてください。アプリやゲームが共有スペースまたはフルスペースで実行されているとき、ユーザが開く最初のウインドウやボリュームは、自動的にユーザの正面の便利な場所に配置されます。フルスペースで実行中の場合は、3Dコンテンツを適切に配置するために、アプリやゲームがユーザの頭の向きに関する情報へのアクセスをリクエストすることもできます。どんな場合でも、広い領域で視線をあちこち動かしたり、いくつもの奥行きを素早く行き来したりする必要がないデザインにすることで、体験の視覚的な快適性を高めることができます。ガイダンスは[奥行き](/jp/design/human-interface-guidelines/spatial-layout#Depth)を参照してください。

**快適に見ることができる距離にコンテンツを配置する。** 例えば、コンテンツの閲覧や操作を長時間にわたって快適に行えるようにするには、コンテンツをユーザから1m以上離して配置するように心がけてください。コンテンツを至近距離に配置するのは、基本的に、閲覧や操作がほんの短い時間で終わるときだけにすることをおすすめします。

**標準的なUIコンポーネントを使うのが望ましい。** システムが提供するコンポーネントは、ユーザが注視すると一貫した反応を示します。カスタムコンポーネントが標準的でない視覚情報を使って視覚的なフィードバックを返す場合、そのコンポーネントの動作を習得するのが難しくなることがあります。

## [項目を見やすくする](/jp/design/human-interface-guidelines/eyes#Making-items-easy-to-see)

**視覚的に余計なものを最小限に抑える。** 見るものが多いと、目的のオブジェクトを見つけるのが難しくなることがあります。視覚的に動くものは、さらに注意をそらす原因になりかねません。人は、視野の中（特に周辺視野）で動くものを知覚すると反射的にそちらに目を向ける傾向があり、関心のあるオブジェクトを注視し続けるのが難しくなってしまいます。例えば、ユーザが注視しているボタンの近くにコンテンツを表示すると、ユーザの視線がボタンではなく、新しいコンテンツに移ってしまうかもしれません。

**項目の周りに十分なスペースを確保して、項目に簡単に注視できるようにする。** たとえ1つの場所を注視している場合でも人間の目は細かく揺れる性質があるので、UIオブジェクトを詰め込むと、UI間を移動せずに1つだけを注視するのが難しくなってしまいます。インタラクティブな項目と項目の間に十分なスペースを確保するには、項目の境界間のマージンを16ポイント以上にするか、項目の中心間の距離が常に60ポイント以上になるように配置します。レイアウトに関するその他のガイダンスは、[レイアウト](/jp/design/human-interface-guidelines/layout)および[空間レイアウト](/jp/design/human-interface-guidelines/spatial-layout)を参照してください。

**反復的な模様やテクスチャで視野を埋めることは避ける。** 人は、左右の目で模様やテクスチャ内の別々の要素を見てしまうことがあり、そうなった場合、それぞれの要素の奥行きが異なっているように見えます。これを避けるために、模様を使用するのは小さい領域に留めることを検討してください。

## [操作を促す](/jp/design/human-interface-guidelines/eyes#Encouraging-interaction)

**ユーザが操作したいであろう項目への注視を促すには、控えめに視覚的なヒントを与えることを検討する。** 例えば、視野の中心にアイテムを配置する、控えめな動きを付ける、コントラストを上げる、カラーや大きさを変えるなどすると、ユーザの注意を引くのに効果的な場合が多いです。基本的には、大げさではないけれども目には留まるようなヒントを示すのがよいでしょう。

**インタラクティブな項目は基本的に丸みのある形状にする。** 人は図形の角に視線を引っ張られる傾向があるため、角があると項目の中央を見つめ続けるのが難しくなります。項目の形状に丸みがあればあるほど、ユーザは視線を向けやすくなります。

![正方形のボタン。](https://docs-assets.developer.apple.com/published/d60c5b225c91f041c5ef7e273a9219b6/visionos-eyes-sharp-button-incorrect%402x.png)

![丸に囲まれたバツ印が不適切な使い方であることを示しています。](https://docs-assets.developer.apple.com/published/209f6f0fc8ad99d9bf59e12d82d06584/crossout%402x.png)

![円形ボタン。](https://docs-assets.developer.apple.com/published/61afcfc99cebef8a0feae23fc5803edc/visionos-eyes-rounded-button-correct%402x.png)

![丸に囲まれたチェックマークが適切な使い方であることを示しています。](https://docs-assets.developer.apple.com/published/88662da92338267bb64cd2275c84e484/checkmark%402x.png)

**インタラクティブなコンポーネントを複数の要素で構成する場合は、強調表示される図形の中にコンポーネント全体を入れるようにする。** 例えば、画像とラベルを組み合わせて1つのインタラクティブなコンポーネントとして動作させる場合は、両方の要素を囲むカスタム領域を定義して、ユーザがどちらの要素を注視した場合でもその領域全体が強調表示されるようにする必要があります。

## [カスタムのホバーエフェクト](/jp/design/human-interface-guidelines/eyes#Custom-hover-effects)

アプリやゲーム内での必要に応じて、システムが提供するUI要素、カスタムUI要素、RealityKitエンティティなどの要素をユーザが注視すると独自の方法でアニメートするようなホバーエフェクトを設計することができます。カスタムのホバーエフェクトを使用して標準のエフェクトを置き換えたり拡張したりできます。

カスタムのホバーエフェクトの設計を始める前に、その仕組みを理解しておくことが大切です。要素でカスタムのホバーエフェクトを有効にするには、そのエフェクトが表示された状態とそうでない状態という2つの状態（見た目）を作成します。ユーザがアプリやゲームで要素を注視すると、あらかじめ定義されたホバーエフェクトをシステムが適用しますが、このプロセスはソフトウェアのプロセスの外で実行されます。つまり、システムがいつカスタムのホバーエフェクトを適用するか、およびその時点で要素がどんな状態にあるかを知ることはできません。カスタムのホバーエフェクトがプロセス外で実行されるために、ユーザがいつ要素を注視しているかを知る必要があるようなコードは実行できません。

カスタムのホバーエフェクトに可能なことと不可能なことを示す例として、写真ブラウズアプリを考えてみましょう。写真が「お気に入り」に追加されているかどうかによって、その写真のカスタムエフェクトで表示されるシンボルが異なります。写真のカスタムのホバーエフェクトに適切なシンボルをアプリが指定し、ユーザが写真を注視するとシステムがエフェクトを表示します。ですが、ユーザが写真を注視しているという情報をシステムがアプリに伝えることはないため、お気に入りに追加するアクションをホバーエフェクトで実行することはできません。

**カスタムのホバーエフェクトは体験中の特別な瞬間を強調したり高めたりする形で使用する。** ユーザは、視覚的なフィードバック（タブバーやツールチップの場合は追加情報）を提供する標準のホバーエフェクトに慣れているため、カスタムのホバーエフェクトは特に目に留まりやすいです。カスタムのホバーエフェクトを追加しすぎたり、標準のエフェクトで十分なときに使用したりすると、デザインのインパクトを薄め、タスクからユーザの気を逸らし、さらには視覚的な不快感を与える可能性もあります。

**適切な遅延時間を選択する。** 要素のカスタムのホバーエフェクトは、ユーザにその要素をどのように操作してほしいかに応じて、遅延なしで表示されるもの、短い遅延のあとに表示されるもの、やや長い遅延のあとに表示されるもののいずれかを選択できます。

  * **遅延なし（デフォルト）。** 遅延なしで表示されるカスタムのホバーエフェクトは、エフェクトが目立たない場合や、ノブがスライダに表示されるときのように操作を求める場合に特に有用です。

  * **短い遅延。** ユーザが要素を注視して、エフェクトが表示されるのを待たずに素早く操作できるようにする場合は、短い遅延を使用することを検討してください。例えば、タブバーのタブの拡大がこれに該当します。

  * **長い遅延。** カスタムのホバーエフェクトに追加情報が表示される場合（ボタンの下に表示されるツールチップなど）は、やや長い遅延が良いでしょう。ほとんどのユーザは追加情報を毎回確認する必要はないためです。

**要素のプライマリビューのうちカスタムのホバーエフェクトの両方の状態で変化しないものを1つ以上残しておく。** ホバーエフェクトのアニメーション中に少なくとも1つのプライマリビューを固定しておくと、視覚的に安定し、ユーザが要素の遷移を追いやすくなります。カスタムのホバーエフェクト中に要素のすべてのビューが移動または変化すると、ユーザが混乱し、何が起きているのか分かりづらくなります。

**カスタムのホバーエフェクトは入念にテストする。** カスタムのホバーエフェクトが、ユーザの気を逸らすことなく、見栄えが良く、適切に反応し、体験を生き生きとしたものにできているかどうかを判断するには、テストするしかありません。カスタムのホバーエフェクトをどう使用すれば体験が高まるかの感覚をつかめるように、カスタムのホバーエフェクトをテストするときにはApple Vision Proを装着してください。
