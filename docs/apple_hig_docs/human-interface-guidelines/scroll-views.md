# スクロールビュー

スクロールビューは、ビューに収まりきらないコンテンツを縦または横にスクロールして表示できるようにする機能です。

![スクロール可能な図案化された画像ビュー。画像は全体が、6色のオリジナルAppleロゴを連想させる赤色になっています。](https://docs-assets.developer.apple.com/published/9db7044c1da1daa9ba165dc76811fd2e/components-scroll-view-intro%402x.png)

スクロールビュー自体に見た目という概念はありませんが、半透明の _スクロールインジケータ_ を表示できます。これは通常、ユーザがビューのコンテンツをスクロールし始めたときに表示されます。スクロールインジケータの見た目と動作はプラットフォームによって異なる可能性がありますが、すべてのインジケータはスクロールアクションについての視覚的なフィードバックを提供します。例えば、iOS、iPadOS、macOS、visionOS、watchOSのインジケータでは、現在表示されているコンテンツがビューの先頭近くか、中央か、末尾かが示されます。

## [ベストプラクティス](/jp/design/human-interface-guidelines/scroll-views#Best-practices)

**デフォルトのスクロールジェスチャとキーボードショートカットに対応する。** ユーザはシステム全体で採用されているスクロール動作に慣れており、どこでも同じように動作すると考えるのは当然です。カスタムのスクロール機能を作成してビューに適用する場合でも、スクロール操作でユーザが戸惑うことがないよう、柔軟に対応してください。

**コンテンツをスクロールできることを明示する。** スクロールインジケータは常に表示されるわけではありません。コンテンツがビューより大きい場合は、それを明確に示すことがユーザの助けになります。例えば、ビューの端にコンテンツの一部を表示すれば、その方向にまだコンテンツがあることが分かります。ほとんどのユーザには、まだコンテンツがあるかどうかを知ろうとするスクロール操作が染み着いていますが、それが一目で分かるようにしておくのは親切な設計といえます。

**スクロールビューの内部に同じ方向の別のスクロールビューを配置しない。** 同じ方向のスクロールビューを入れ子にすると、操作しにくい予想外のインターフェイスができることがあります。縦方向のスクロールビューの内部に横方向のスクロールビューを配置すること（またはその逆）は問題ありません。

**コンテンツに適合する場合はページ単位のスクロールに対応する。** 場合によっては、連続的にスクロールするより一定の高さや幅でスクロールする方がユーザにとって便利です。ほとんどのプラットフォームでは、そのような _ページ_ のサイズ（通常はビューの現在の高さまたは幅）を定義して、一度に1ページずつスクロールする操作を提供できます。ページ単位でスクロールする際、ページの最後に次のページとオーバーラップして表示する部分（1行のテキスト、1列のグリフ、画像の一部など）を設けると、ユーザがコンテキストを把握し続ける助けになります。デベロッパ向けのガイダンスは、[`PagingScrollTargetBehavior`](/documentation/SwiftUI/PagingScrollTargetBehavior)を参照してください。

**場合によっては自動的にスクロールしてユーザが現在の位置を把握できるようにする。** ほとんどのスクロール操作はユーザが開始しますが、コンテンツの目的の位置がビューの範囲から外れてしまった状況で、自動スクロールがユーザの助けになることがあります。以下のような状況が考えられます:

  * ビューに現在表示されていない位置で、アプリがコンテンツの選択や挿入ポイントの設定などの処理を実行するとき。例えば、ユーザが検索しているテキストが見つかったら、コンテンツを自動スクロールしてその場所を選択状態にし、ビューに表示できます。

  * 現在表示されていない位置でユーザが入力操作を開始したとき。例えば、あるページに挿入ポイントを置いたままユーザが別のページに移動した場合、ユーザがテキストを入力し始めると同時にその挿入ポイントまで自動的に戻ることができます。

  * ユーザがビューの境界をまたいでポインタを移動させる選択操作を行ったとき。この場合は、ポインタが移動した方向に自動スクロールしてポインタに追従します。

  * ユーザが何かを選択し、新しい場所にスクロールしてから選択部分に対する操作を行ったとき。この場合は、その選択部分がビューに表示されるまで自動スクロールしてから操作を実行します。

いずれの場合も、ユーザがコンテキストを見失わないようにするのに必要な分だけ自動的にスクロールしてください。例えば、選択部分が一部表示されていれば、選択部分の全体がビューに表示されるまでスクロールする必要はありません。

**拡大/縮小に対応する場合は適切な最大および最小倍率を設定する。** 例えば、1文字で画面がいっぱいになるまでテキストを拡大しても、ほとんどの場合意味がありません。

## [スクロールエッジエフェクト](/jp/design/human-interface-guidelines/scroll-views#Scroll-edge-effects)

iOS、iPadOS、macOSでは、ぼかしの強さを変化させる _スクロールエッジエフェクト_ により、コンテンツ領域と[Liquid Glass](/jp/design/human-interface-guidelines/materials#Liquid-Glass)コントロール領域（[ツールバー](/jp/design/human-interface-guidelines/toolbars)など）の境界を表現しています。固定要素がスクロール可能な要素と重なる場合、ほとんどの状況でスクロールエッジエフェクトが自動的に適用されます。カスタムのコントロールやレイアウトを使用する場合、このエフェクトが表示されないことがあります。その場合は、手動で追加する必要があります。デベロッパ向けのガイダンスは、[`ScrollEdgeEffectStyle`](/documentation/SwiftUI/ScrollEdgeEffectStyle)および[`UIScrollEdgeEffect`](/documentation/UIKit/UIScrollEdgeEffect)を参照してください。

スクロールエッジエフェクトには、softとhardという2つのスタイルがあります。

  * ほとんどの場合、特にiOSとiPadOSでは、ツールバーやインタラクティブ要素（ボタンなど）に最適な[`soft`](/documentation/SwiftUI/ScrollEdgeEffectStyle/soft)エッジエフェクトを使用してください。このエフェクトは繊細な遷移を実現します。

  * 主にmacOSでは、より強くて不透明な境界線を実現する[`hard`](/documentation/SwiftUI/ScrollEdgeEffectStyle/hard)エッジエフェクトを使用してください。インタラクティブなテキスト、背景のないコントロール、または明確さを必要とする固定テーブルヘッダに最適です。

**スクロールエッジエフェクトはスクロールビューがフローティングインターフェイス要素に隣接している場合にのみ使用する。** スクロールエッジエフェクトは装飾ではありません。オーバーレイのように画面がブロックされたり、暗くなったりすることはなく、コントロールとコンテンツの境界を明示するために存在しています。

**スクロールエッジエフェクトは1つのビューにつき1つだけ適用する。** iPadとMacのSplit Viewレイアウトでは、パネルごとにスクロールエッジエフェクトをかけることができます。その場合は、同じ高さにして配置に一貫性を持たせるようにします。

## [プラットフォームの考慮事項](/jp/design/human-interface-guidelines/scroll-views#Platform-considerations)

### [iOS/iPadOS](/jp/design/human-interface-guidelines/scroll-views#iOS-iPadOS)

**スクロールビューがページ単位モードのときはページコントロールの表示を検討する。**[ページコントロール](/jp/design/human-interface-guidelines/page-controls)を使うと、ページや画面などの単位でコンテンツがいくつに分割されているか、現在表示されているのはどれかが分かります。例えば「天気」では、ユーザが保存した複数の場所を切り替える際にページコントロールが使用されています。スクロールビューでページコントロールを表示する場合は、コントロールが重複してユーザが混乱しないように、ページコントロールと同じ方向のスクロールインジケータを表示しないでください。

### [macOS](/jp/design/human-interface-guidelines/scroll-views#macOS)

通常、macOSでは、 _スクロールインジケータ_ は _スクロールバー_ と呼ばれます。

**必要に応じて、パネルではsmallまたはminiのスクロールバーを使う。** スペースに余裕がない場合は、ほかのウインドウと共存する必要があるパネルで小さいスクロールバーを使用できます。このようなパネルでは、必ずすべてのコントロールのサイズを合わせてください。

### [tvOS](/jp/design/human-interface-guidelines/scroll-views#tvOS)

tvOSのビューはスクロールできますが、スクロールインジケータを備えた個別のオブジェクトとしては扱われません。代わりに、コンテンツのサイズが画面より大きい場合は、操作対象項目が常に表示されるようにインターフェイスが自動的にスクロールされます。

### [visionOS](/jp/design/human-interface-guidelines/scroll-views#visionOS)

visionOSのスクロールインジケータは小さな固定サイズで、ユーザは、大きく動かさずに効率的にスクロールできることがすぐに分かります。スクロールインジケータは、ユーザが見つけやすいように、常にウインドウとの関係で予想できる位置に表示されます。縦にスクロールしているときは末尾側の縦中央に、横にスクロールしているときはウインドウの下端の横中央になります。

ユーザが、スクロールしたい方向にコンテンツをスワイプし始めると、スクロールインジケータがウインドウの端に現れ、ジェスチャの効果が視覚的に強調されます。また、コンテンツの現在の位置と全体の長さが分かるフィードバックが示されます。ユーザがスクロールインジケータを見ながらドラッグジェスチャを開始すると、インジケータがジョグバーのように働いて、コンテンツの位置ではなくスクロール速度を調整できるようになります。この動作では、スクロールインジケータに、ユーザがジェスチャを細かく調整するのに応じて速度が上下する目盛りが表示されます。ユーザはこの視覚的なフィードバックを見ながら、スクロールの速さを正確にコントロールできます。

以下のカスタムコントロールを使用できます 

このビデオコンテンツの説明: メモアプリの長いページにスクロールインジケータが表示されている録画。ユーザがページを素早くドラッグすると、インジケータにはスクロール速度に合った目盛りが表示されます。 

再生 

**必要に応じて、スクロールインジケータのサイズを考慮する。** インジケータ全体のサイズは小さいものの、iOSのコンポーネントよりは少し厚みがあります。コンテンツのマージンが狭い場合は、コンテンツがスクロールインジケータと重ならないように、マージンを増やすことを検討してください。

### [watchOS](/jp/design/human-interface-guidelines/scroll-views#watchOS)

**コンテンツはなるべく縦にスクロールするようにする。** Apple Watchのユーザは、アプリ間やアプリ内のナビゲーションにDigital Crownを使用することに慣れています。アプリに含まれているリストやコンテンツビューが1つだけの場合、アプリのコンテンツが画面の高さよりも長いときにDigital Crownを回すと縦にスクロールします。

**ページ単位でスクロールできるようにするにはタブビューを使用する。** watchOSでは、タブビューがページとして表示されます。縦方向スタックにタブビューを配置すると、Digital Crownを回してコンテンツのフルスクリーンページ間を縦に移動できます。この場合、Digital Crownの横に表示されるページインジケータによって、現在のページ内のどこにいるか、またページセット内のどこにいるかが分かります。ガイダンスは、[タブビュー](/jp/design/human-interface-guidelines/tab-views)を参照してください。

**ページ形式のコンテンツを表示する場合は、各ページのコンテンツを画面の高さに収まる量に限定することを検討する。** この制約を設けることで、個々のページの目的が明確になり、より一層情報を読み取りやすいデザインに仕上がります。アプリのページが画面の高さよりも長くなった場合でも、Digital Crownを使えば短いページ間のナビゲーションも長いページのコンテンツのスクロールも可能です。必要に応じてページインジケータが拡大し、スクロールインジケータになるからです。高さが可変のページはよく考えて使い、可能であれば、高さが固定のページのあとに配置してください。
