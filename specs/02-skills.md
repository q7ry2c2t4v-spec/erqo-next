# スキル一覧

スキルの詳細な手順は各 SKILL.md に記載されている。ここでは「いつ何を使うか」の判断基準を示す。

## メインフローのスキル（プロジェクトワークフロー）

`/dsgn → /tp → /codi` の 3 フェーズで構成されるプロジェクト向け開発フローを、本 OS では通称 **プロジェクトワークフロー** と呼ぶ（定義の一次ソース: `specs/01-workflow.md`）。以下はその 3 フェーズを構成するスキル。

**対象:** 3 スキルとも **プル子・プタ子両方** で使える (Next.js 非依存)。各 `skills/<name>/SKILL.md` のフロントマターに `target: project` と宣言されており、本元 (エル子) 側では `install.py:_scan_skills` がこれを検出して機械除外するため、エル子では起動できない（エル子は「システム調整フロー」で動く。`specs/01-workflow.md` 参照）。`/codi` のレイアウトタスク (`LAYOUT-*` / `PAGE-*` / `CLONE-*` / `DESIGN-*`) は `/layo` との併用前提なので、`/layo` 非配布のプタ子では実質使えない（通常タスクは問題なく使える）。

### /dsgn — 対話・設計

- **いつ:** プロジェクト初期に1度
- **目的:** 大枠の機能セクション一覧を決める
- **成果物:** `.libs/design/project-spec/`
- **注意:** 各セクションの詳細には踏み込まない

### /tp SECTION — タスク計画

- **いつ:** セクションに着手する時
- **目的:** 詳細設計 + タスク分解 + 識別コード・依存関係付与
- **成果物:** `.libs/design/<section>/`
- **前提:** /dsgn で対象セクションが定義済みであること
- **コンテキスト収集:** `core/index.py collect` で既存の関連ページを自動収集

### /codi ID-CODE — 実装パイプライン

- **いつ:** タスクを実装する時
- **目的:** 5 ステップの自動化パイプライン（準備 → 実装 → 検証 → 記録 → コミット）
- **タスク分岐:** ステップ2（実装）はタスク種別で分岐する。通常タスク (2-A) は直接実装。レイアウトタスク (`LAYOUT-*` / `PAGE-*` / `CLONE-*` / `DESIGN-*`) は 2-B で `core/clone.py` の **7 サブステップ** (recon → dump → apply → rules → build → assemble → baseline) に展開される
- **成果物:** 実装コード + 記録ページ + ユーザーガイド + コミット
- **前提:** /tp で対象タスクの識別コードが付与済みであること
- **状態管理:** `core/state.py` がパイプライン状態を管理。中断後の再開可能。レイアウトタスクは `layout_substep` でサブステップ進捗も追跡
- **依存チェック:** 開始時に依存タスクの完了状態を自動チェック

## 補助スキル

補助スキルは以下 7 つ。エル子 (本元) / プル子 (Next.js プロジェクト) / プタ子 (任意スタックプロジェクト) のどれで使えるかを列で示す。`✓` = 使える / `-` = 使えない (配布されない)。

| スキル | エル子 | プル子 | プタ子 | 役割 |
|---|---|---|---|---|
| `/layo` | - | ✓ | - | レイアウトデザインの策定・修正 (Next.js 特化、何度でも実行可能) |
| `/rsrc` | ✓ | ✓ | ✓ | 技術情報のウェブリサーチ → `.libs/research/` に保存 |
| `/wrap` | ✓ | ✓ | ✓ | セッションログ + git コミットを 1 コマンドで |
| `/ret`  | ✓ | ✓ | ✓ | 前回セッションのコンテキスト復元 |
| `/aud`  | ✓ | ✓ | ✓ | 作業監査 (整合性・残骸・依存・重複チェック) |
| `/ruru` | - | ✓ | ✓ | プロジェクトルールの登録・更新 (エル子では `specs/` を直接編集するため不使用) |
| `/fb`   | - | ✓ | ✓ | erqo-next へのフィードバック送信 (エル子は自分自身なので不要) |

配布は各 SKILL.md の frontmatter `target:` (エル子/プロジェクト境界) と `variant:` (プル子/プタ子境界) を `install.py:_scan_skills` が機械解釈して決める。

### /layo ID-CODE — レイアウトデザイン策定

- **いつ:** レイアウトのデザインを決めたい時・修正したい時
- **目的:** 参考サイトの解析 + 要望の反映で本棚ページを作成・更新する
- **成果物:** `.libs/storybook/<slug>/` に本棚ページ + 取材データ
- **前提:** `/tp` で対象タスクの識別コード (`LAYOUT-*` / `PAGE-*` / `CLONE-*` / `DESIGN-*`) が付与済みであること
- **特徴:** 何度でも実行可能。参考 URL の追加・要望の修正・トークンの調整・部分削除が対話ベースでできる。完成したら `/codi` で実装に進む
- **スクリプト:** `core/layo.py` でコンテキスト収集 (docs 検索 + ルール + 関連ページ)、`core/clone.py` の recon / dump / apply / rules で本棚ページ操作

### /rsrc TOPIC — ウェブリサーチ

- **いつ:** 実装に必要な技術情報を事前に調べたい時
- **目的:** ウェブリサーチを行い、結果を .libs に保存する
- **成果物:** `.libs/research/<topic>/` にリサーチページ
- **特徴:** タグ付きで保存されるため `/tp` の index.py collect から自動参照される

### /wrap — セッション終了統合（旧 /mmy + /gh）

- **いつ:** セッションを終わらせる時 / 作業の節目で記録 + 保存をまとめたい時
- **目的:** セッションログ作成 + git コミットを 1 コマンドにまとめ、承認も 1 回で完結させる
- **成果物:** `.libs/session-logs/YYYY-MM/slog-YYYYMMDD-NNN.md` + git コミット
- **特徴:** ログ案とコミットメッセージ案を **両方並べて** 提示 → y で承認 → 一気に実行。push は行わない
- **スクリプト:** 差分確認に `core/git_ops.py check`。コミットは `git` を直接呼ぶ

### /ruru — プロジェクトルール管理

- **いつ:** ルールを決めた時
- **目的:** プロジェクトルールの登録・更新
- **成果物:** `.libs/rules/` + CLAUDE.md に @import 追加
- **特徴:** ルール内容からスコープ（`適用: ui/api/all`）を自動推定。load.py のスコープフィルタリングと連携

### /ret — セッション復元

- **いつ:** 前回の続きをしたい時、キーワードが不明な時
- **目的:** 直近セッションログから前回コンテキストを取得
- **成果物:** なし（読むだけ）
- **特徴:** セッションログのタグから `index.py collect` を自動実行し、関連コンテキストを拡充

### /aud — 作業監査

- **いつ:** 作業の節目 / `/wrap` の直前 / リファクタ・削除作業の後
- **目的:** セッション中の変更箇所の整合性・古いコード残骸・依存関係の食い違い・重複・仕様との齟齬を軽量チェックする
- **成果物:** なし（レポート提示のみ、修正はユーザー承認後に実行）
- **特徴:** `/ret` と同じ「読むだけ」系。プロジェクト全体ではなく今セッションで触った範囲のみを対象にする。所見は「問題なし / 注意 / 要対応」の 3 段階で分類

### /fb — フィードバック送信

- **いつ:** erqo-next への改善提案・エラー報告を送りたい時
- **目的:** プロジェクトから erqo-next 本体へフィードバックを送信する
- **成果物:** `.claude/state/feedback/` に JSON レポート
- **スクリプト:** `skills/fb/handler.py`

## スキルの選び方

```
セッション開始
  ├── 前回の続き → /ret
  ├── 新規プロジェクト → /dsgn
  ├── 技術情報を調べたい → /rsrc TOPIC
  ├── セクション着手 → /tp SECTION
  ├── レイアウトデザイン策定 → /layo ID-CODE
  ├── タスク実装 → /codi ID-CODE
  ├── ルール追加 → /ruru
  ├── 作業の整合性チェック → /aud
  ├── フィードバック送信 → /fb
  └── セッション終了 → /wrap (ログ + コミットを一括)
```
