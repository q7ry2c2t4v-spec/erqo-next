# 開発フロー

フローは**プロジェクト向け**と**本元リポジトリ向け**で異なる。まず視点を確定してからどちらのフローを適用するか決める（視点判断は `specs/07-scopes.md`）。

## プロジェクトワークフロー（プロジェクト向け: 3 フェーズ開発フロー）

`/dsgn → /tp → /codi` と続くプロジェクト向け 3 フェーズ開発フローを、本 OS では通称 **プロジェクトワークフロー** と呼ぶ。以降この略称を各ドキュメントで使う。


```
1. /dsgn  対話・設計    大枠の設計書を策定（セクション一覧）
              ↓
2. /tp    タスク計画    セクションごとに詳細設計 + タスク分解
              ↓
   /layo  デザイン策定  レイアウトタスクの場合、デザイン設計書を作成 (プル子のみ)
              ↓
3. /codi  実装パイプライン  タスク単位で 準備→実装→検証→記録→コミット
              ↓
         (2 に戻る — 次のセクション or タスクへ)
```

プロジェクトではこのフローでアプリケーションを作っていく。`/layo` はレイアウトタスク (`LAYOUT-*` / `PAGE-*` / `CLONE-*` / `DESIGN-*`) の場合のみ実行する。

**プル子 / プタ子 の差分:**

| スキル | プル子 (`variant: nextjs`) | プタ子 (`variant: generic`) |
|---|---|---|
| `/dsgn`, `/tp`, `/codi` | 使える | 使える (Next.js 非依存のため) |
| `/layo` | 使える (Next.js 特化装備) | **配布されない** |
| `/codi` のレイアウトタスク | 使える | `/layo` が無いので結果的に使えない |
| 補助スキル (`/rsrc` `/wrap` `/ret` `/aud` `/ruru` `/fb`) | 使える | 使える |

スキル配布は各 SKILL.md の frontmatter `variant:` と、プロジェクトの `.claude/state/variant.json` の印で機械的に振り分けられる (`core/install.py:_scan_skills`)。

### フェーズ1: 設計（/dsgn）

プロジェクト初期に **1度だけ** 実行する。

- ユーザーと対話してプロジェクトの目的・全体像を整理する
- 機能セクション一覧を決める（深入りしない）
- 成果物: `.libs/design/project-spec/` に設計概要ページ

**次へ進む条件:** セクション一覧が決まったらフェーズ2へ

### フェーズ2: タスク計画（/tp SECTION）

セクションに取り掛かるたびに実行する。

- 対象セクションの詳細をユーザーと対話して詰める
- タスクに分解し、各タスクに識別コードと依存関係を付与する
- 成果物: `.libs/design/<section>/` にタスク計画ページ

**次へ進む条件:** タスク分解が完了したらフェーズ3へ

### フェーズ3: 実装（/codi ID-CODE）

タスクごとに実行する。5 ステップの自動化パイプライン。

1. **準備** — タスク読み込み + UI 判定 + コンテキスト収集
2. **実装** — 承認フロー後に実装。タスク種別で分岐:
   - **2-A 通常タスク (UI / API / その他)** — 収集したコンテキストに従って直接実装
   - **2-B レイアウトタスク (`LAYOUT-*` / `PAGE-*` / `CLONE-*` / `DESIGN-*`)** — `core/clone.py` の **7 サブステップ** (recon → dump → apply → rules → build → assemble → baseline) に自動展開。`state.py` の `layout_substep` で進捗管理。詳細は `skills/codi/SKILL.md` 2-B
3. **検証** — 自動品質チェック（tsc, eslint, playwright, lighthouse 等）
4. **記録** — features ページ作成 + TP ステータス更新 + ユーザーガイド更新
5. **コミット** — 全変更を1コミットにまとめる

**次へ進む条件:** セクション内の全タスク完了 → フェーズ2に戻り次のセクションへ

## 本元リポジトリ向け: システム調整フロー

本元リポジトリ（IS_SOURCE=True）では `/dsgn`, `/tp`, `/codi` は**対象外**。`constants.py` の `SOURCE_EXCLUDED_SKILLS` がこの 3 スキルを本元から除外する。

本元での作業は「エル子 OS 自身のシステム調整」であり、次のシンプルなフローで進める:

```
1. 対話・設計    ユーザーと変更方針を合意（承認フロー: specs/00-identity.md）
        ↓
2. 直接編集     specs/, core/, skills/ を直接編集する（読み取り専用の制約なし）
        ↓
3. 動作確認     必要なら python コマンド等で手動検証
        ↓
4. /wrap        セッションログ + git コミットを一括で実行（記録と保存）
```

### 本元フローの原則

- **設計書やタスク分解は作らない:** システム調整は小規模で、`.libs/design/` を介すると overhead になる。対話で合意 → 直接編集する
- **記録は `/wrap` のセッションログに集約:** `.libs/features/` への記録も不要。「何を変えたか / なぜ変えたか / 決定事項」はセッションログに書く
- **自動品質チェック（tsc/eslint/playwright 等）は対象外:** 本元リポジトリは Python スクリプトと Markdown が中心で、Next.js プロジェクト向けのチェックとは目的が違う
- **`giget` 経由での配布が前提:** 本元で加えた変更は `core/`, `skills/`, `specs/` として各プロジェクトに配布される。プロジェクト側で動くことを念頭に置いて編集する
- **コード編集前にコーディングルールを必ずロード:** `core/`, `skills/`, `scripts/` 配下のコードファイル (`.py` / `.ts` / `.tsx` 等) を編集する前に `python core/coding_rules.py <対象ファイル>` を実行し、返ってきたルールファイル (L1 ハブ + 該当 L2 / L3) を**全て Read してから編集に入る**。判定は `core/coding_rules.py` が機械的に行うため AI の自己判断は挟まない (`specs/06-coding-rules.md` §0 / `specs/08-responsibility.md`)。`PreToolUse` Hook の `coding_rules_hook.py` が編集直前に該当ルールを stderr に通知するため、Hook 出力を見落とさない

### 本元で使えるスキル一覧

本元 (エル子) で使えるスキルは各 `skills/<name>/SKILL.md` の frontmatter `target:` で機械的に決まる (`target: project` なら `install.py:_scan_skills` が本元側で除外、`target: both` なら両方で使える)。現在のエル子で使える補助スキル:

- `/rsrc` — 技術情報のリサーチ
- `/wrap` — セッションログ + コミット
- `/ret` — 前回セッションの復元
- `/aud` — 作業監査

除外 (`target: project`) されているスキル: `/dsgn`, `/tp`, `/codi`, `/ruru`, `/fb`

## 補助スキル（フローとは独立）

| スキル | 役割 | いつでも使える |
|---|---|---|
| `/rsrc` | ウェブリサーチ → .libs に保存 | はい |
| `/wrap` | セッションログ + git コミットを1コマンドで（旧 /mmy + /gh の統合版） | はい |
| `/ruru` | プロジェクトルールの登録・更新 | はい |
| `/ret` | 前回セッションのコンテキスト復元 | はい |
| `/aud` | 作業監査（整合性・残骸・依存・重複チェック） | はい |
| `/fb` | erqo-next へのフィードバック送信 | はい |

これらはフェーズに依存しない。ユーザーが任意のタイミングで使う。
