# プロジェクトガイドとユーザーガイド

プロジェクトには2つの文書体系がある:

| 文書 | 読者 | 目的 |
|---|---|---|
| `.libs/` | AI・開発者 | 復元可能レベルの技術詳細バックアップ |
| `.ui-specs/` | 誰でも | 何ができてどう使うかの要約（専門用語なし） |

---

## .libs/ — 開発知識のバックアップ

### 本棚形式（3層構造）

```
.libs/                  ← ルート
 └ 棚（フォルダ）       ← ジャンル
    └ 本（フォルダ）     ← トピック
       └ ページ（.md）  ← 具体的な情報
```

### 棚構成

| 棚 | 用途 | 作成元 |
|---|---|---|
| `design/` | 大枠設計・タスク計画 | /dsgn, /tp |
| `features/` | 実装済み機能の詳細記録 | /codi |
| `docs/` | 外部ドキュメント参照 | 手動配置 |
| `research/` | ウェブリサーチの調査結果 | /rsrc |
| `rules/` | プロジェクトルール | /ruru |
| `session-logs/` | セッションログ | /mmy |
| `archive/` | アーカイブ | 自動 |

### 復元可能性の原則

**コードが全て消失しても、.libs の情報だけで完全に復元できるレベルの詳細さを保つ。**

記録すること:
- 実装した機能の仕様・構成
- なぜその設計にしたか（設計判断・トレードオフ）
- 遭遇したトラブルと解決策
- 今後の構成予定・修正案

### 識別コード

すべてのページに識別コードを付与する。

**形式:** `カテゴリ-トピック-内容`（英大文字 `A-Z`、数字 `0-9`、ハイフン `-`）

**ファイル名:** 識別コードを小文字にしたもの + `.md`

### ページフォーマット

```markdown
# 識別コード — ページタイトル

関連: 関連ページの識別コード1, 識別コード2
タグ: 日本語タグ1, 英語タグ1

## 識別コード.セクション名 — セクションタイトル
本文...
```

- 関連ページの記述は片方向でよい（検索は双方向で探す）
- 1ページは170行前後を目安とし、超える場合は分割

### インデックス（_index.json）

全ページのメタデータを `_index.json` に集約。`core/index.py reindex` で再生成する。

配置先: `.claude/state/_index.json`

## .ui-specs/ — ユーザー向け機能要約

### 目的

.libs とは別の読者層を想定する。開発者でない人が読んでもわかる機能ガイド。

### 構成

```
.ui-specs/
├── index.md      ← 全体目次（スクリプトが自動生成）
├── auth.md       ← セクション単位で1ファイル
├── api.md
└── ...
```

### 執筆ルール

- **専門用語禁止** — コード名、ライブラリ名、技術用語を使わない
- **要約レベル** — 「何ができるか」「どう使うか」に焦点。内部構造には触れない
- **平易な日本語** — 開発者でない人が読んでもわかる表現

### 更新タイミング

/codi ステップ4 (recording) で自動更新される。タスク完了のたびに該当セクションが最新化される。
