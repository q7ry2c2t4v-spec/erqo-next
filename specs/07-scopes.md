# 設定スコープのルール

エル子の OS には **「PC 個人」「マシンごと」という概念は存在しない**。
全ての設定・ファイル・スクリプトは次の 3 つのスコープのいずれかに属する。

## 3 つのスコープ

| スコープ | 適用範囲 | 物理的な置き場所 |
|---|---|---|
| **本元** | 本体リポジトリ（IS_SOURCE）でだけ有効 | `<source>/.claude/`, `<source>/specs/`, `<source>/release.sh` 等 |
| **プロジェクト** | エル子を使う各プロジェクトでだけ有効 | `<project>/.claude/`, `<project>/.libs/`, `<project>/.ui-specs/` 等 |
| **共通** | 本元・プロジェクト両方で有効 | `core/`, `skills/`, `specs/` (giget で配布) + `core/constants.py` の共通定数 |

## 禁止: PC 個人スコープ

`~/.claude/settings.json` のような **PC 個人グローバル設定** に erqo-next 関連の設定を書いてはいけない。

理由:
- ユーザーは複数のデバイスでこの OS を使う
- PC ごとに設定がバラバラだと、デバイス間で挙動が変わって混乱する
- 個人 PC 設定は git で管理されないので、他デバイスや他ユーザーと共有できない

例外: Claude Code 自身の純粋な個人嗜好（テーマ、エディタモード等）は OS と無関係なので `~/.claude/settings.json` に書いてよい。

## 共通スコープの実装方法

「本元・プロジェクト両方で同じ設定」を実現するには:

1. **単一の出どころを `core/constants.py` に定義する**
   - 例: `PERMISSION_DEFAULTS_ALLOW`, `PERMISSION_DEFAULTS_MODE`
2. **`core/install.py` がプロジェクト側に書き込む**
   - 例: `setup_permission_defaults(project_root)`
   - プロジェクトインストール時 / アップデート時に自動適用
3. **`core/dev.py` が本元側に書き込む**
   - 例: `cmd_setup()` 内から同じ `setup_permission_defaults` を呼ぶ
4. **両方とも constants.py の同じ定数を読む**
   - 「同じ場所が出どころ」になるので、変更は 1 か所だけで済む

## デフォルトスコープ

ユーザーから機能追加・修正の依頼があったとき、特に指定がなければ
**「共通スコープ」を前提**にする。

- ユーザーが「本元だけ」「プロジェクトだけ」と明示した場合のみ、その範囲に絞る
- 判断に迷う場合はエル子は必ずユーザーに確認する（曖昧判断・自己判定の禁止 / `specs/08-responsibility.md`）
- 共通実装では「`core/constants.py` に出どころ + `install.py` / `dev.py` から配置」のパターンを使う

## 作業内容からのスコープ判定

スコープは作業の種類から機械的に決まる。視点を宣言した後もこの判定を裏切らない。

| 作業の種類 | スコープ | 実施場所 |
|---|---|---|
| システム的な調整（`core/`, `skills/`, `specs/`, Hook, 共通スクリプト等の編集・追加） | 共通（または本元） | **本元 OS** |
| プロジェクト固有の拡張機能（そのプロジェクトだけで使う hook・追加スキル等） | プロジェクト | プロジェクトの `.claude/` や `CLAUDE.md` |
| プロジェクトの実コンテンツ（`.libs/`, `.ui-specs/`, アプリケーションコード） | プロジェクト | プロジェクト OS |

**原則:** プロジェクト OS 側でシステム的な調整（`/codi` の工程変更・新スキル追加・`core/` の変更等）を行うことは基本的にない。やるなら本元 OS 側で共通スコープとして行い、giget 配布でプロジェクトに届ける。

**判断の順序:**

1. まず作業の種類を見る（システム調整 / プロジェクト拡張 / プロジェクト実コンテンツ）
2. 種類からスコープが決まる
3. スコープから実施場所（本元 or プロジェクト）が決まる
4. 「プロジェクトで試作した方が…」のような視点をまたぐ提案は、システム調整タスクにおいては原則禁止

## 外部サーバー操作のプロジェクト境界

プロジェクト OS で MCP 経由に外部サーバー（Cloudflare / GitHub / Vercel / Supabase 等、**種類を問わない**）を操作する時、**他のプロジェクトのデータに一切触れてはいけない**。

**機械的な判定源:** `.libs/design/project-spec/servers.md`（識別コード `PROJECT-SERVERS`）

このページには各サービスごとに「アカウント識別子 / このプロジェクトで使うリソース一覧（Worker 名・DO namespace・R2 bucket・KV namespace・repo・team 等）/ 用途」が書かれる。ページ自体は `/dsgn` または対話 → 承認 → 編集の流れで更新する（`skills/dsgn/SKILL.md`）。

### 操作前の義務

1. MCP 経由の外部サーバー操作を行う直前に、必ず `PROJECT-SERVERS` ページを Read する
2. 操作対象のリソースがページに記載されているか機械的に照合する
3. **記載されているリソースだけを操作する**
4. 記載がない / ページ自体が存在しないサービスは操作禁止（ユーザーに確認して停止）

### 禁止事項 [MUST]

- ページに記載されていないリソースへの読み取り・書き込み・削除など全ての操作
- 「今回は安全だから」「一覧を見るだけだから」という自己判断で例外を作ること（`specs/08-responsibility.md` 曖昧判断禁止と整合）
- AI がページを勝手に書き換えて記載範囲を広げること（更新は必ず対話 → 承認 → 編集）

### 本元 OS では不要

本元 OS は MCP 経由で外部サーバーを操作しない（`specs/01-workflow.md` 本元フロー原則）。本元には `.libs/design/project-spec/servers.md` は存在せず、このルールは発動しない。

## 違反時の対応

`~/.claude/settings.json` に erqo-next 関連の設定を書いた箇所を見つけたら:

1. その設定を `core/constants.py` の共通定数に移す
2. `setup_permission_defaults` のような共通設定の適用関数経由で配置する
3. PC 個人ファイルからは削除する

新規に設定を追加する場合も、最初から「本元 / プロジェクト / 共通」のいずれに属するかを判断してから配置する。
