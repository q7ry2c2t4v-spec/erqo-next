# 責任分担ルール — タスク管理と作業の分離

## 原則

エル子 OS では **タスク管理は python スクリプトが担う**。
エル子（AI）は **スクリプトの指示に従って作業するだけ**。
判断・工程制御・状態管理を AI が自分で行ってはならない。

## 3 つの禁止事項

### 1. 工程省略の禁止

AI が「この工程は今回必要ない」「スキップしていい」と自己判断して
工程を飛ばすことを禁止する。工程の実行・スキップは全て python スクリプトが制御する。

### 2. 曖昧判断の禁止

「今回は〜だから〜する」というその場限りの判断を禁止する。
判断基準は事前にルール化し、python スクリプトが機械的に判定する。

### 3. 発動条件の自己判定の禁止

「この工程を実行するか？」「この機能が必要か？」を AI が自分で決めない。
必ず python スクリプトが機械的に判定する。
AI から情報が必要な場合は、スクリプトが明示的な受取口
（例: `state.py mark_xxx`）を用意する。

## 役割分担

| 担当 | 内容 |
|---|---|
| python スクリプト | タスク管理・工程制御・状態管理・ファイル操作・パス決定・インデックス・発動条件判定 |
| エル子 (AI) | 実装、記録の執筆、リサーチの執筆、ユーザーガイドの要約 — **内容を作る作業のみ** |

## 新機能設計時の自問チェックリスト

新しいスキル・工程を設計するとき、必ず以下を確認する:

1. **この工程の発動条件は誰が判定するか？** → python スクリプトでなければ NG
2. **AI が「今回は〜だから省略する」と判断する余地はあるか？** → あれば NG
3. **工程の順序・スキップ条件は明文化されているか？** → されていなければ NG
4. **AI が判断材料を提供する場合、その受け渡し方法はスクリプトで定義されているか？** → 未定義なら NG

## AI 記憶への依存禁止

「会話履歴から拾う」「セッション中に触ったものを思い出す」といった
**AI の記憶に依存した判断** は禁止する。情報源は必ず以下のどれかにする:

- python スクリプトの出力
- ファイルシステムの状態
- `git status` / `git diff` の出力
- state.py に記録された状態ファイル

AI の記憶は不確実で、工程ごとに揺らぐ。機械的に取得できる情報源を使う。

## 違反時の対応

既存のスキル・コードに AI 自己判定箇所を見つけたら:

1. 判定ロジックを python スクリプト側に移す
2. AI は結果を受け取って作業するだけにする
3. 判定基準をルール文書化する

## 適用範囲

- 本元リポジトリ、全プロジェクト共通
- 新規スキル・機能の設計時
- 既存スキル・機能の編集時（該当箇所があれば合わせて修正）
