## SLOG-20260411-005.作業内容 — 作業内容

### 1. 視点ブレの失敗と 07 への追記

- 失敗: 視点宣言で「本元 OS」と言った直後に「プロジェクトで試作した方が…」
  とプロジェクト側へ誘導する発言をした
- ユーザー指摘: /codi はシステム的な調整なので本元 OS で行うのが正しい。
  プロジェクト OS ではシステム調整は基本的にない（あっても拡張機能として
  .claude/ や CLAUDE.md に定義する）。これを毎回言わないと分からないのは困る
- 対応: `specs/07-scopes.md` に「作業内容からのスコープ判定」セクションを追加
  - システム調整 / プロジェクト拡張 / プロジェクト実コンテンツ の 3 種類で
    スコープを機械的に判定する表
  - 原則: プロジェクト OS でのシステム調整は基本的にない
  - 判断の順序 4 ステップ（作業種類 → スコープ → 実施場所 → 視点またぎ禁止）

### 2. specs 全体のコンフリクトチェック

specs/00 〜 specs/10 + core/constants.py + core/session.py の動作を確認。
実害ベースで以下を判定:

- **要対応 1 (実害あり):** `.nxt-core/` 記述が本元リポジトリに存在しないフォルダ名で
  統一されていた。本元では `core/` 直下、プロジェクトでは `.nxt-core/` 配下という
  二重構造を specs が反映していなかった
- **要対応 2 (実害あり):** 本元では `/dsgn`, `/tp`, `/codi` が
  `SOURCE_EXCLUDED_SKILLS` で除外されているのに、`01-workflow.md` は
  「メインフロー = dsgn → tp → codi」と書いており、本元での作業フローが
  どこにも明示されていなかった
- **注意 (実害低):** `09-memory.md` の「`.claude/state/` 移行検討中」記述が
  未決定のまま残っている
- **問題なし:** 08 と 10 の AI 判定論は対象領域が違う (工程制御 vs ユーザー対話) ので
  実質整合。PIPELINE_STEPS と 05 の JSON 例も整合。視点宣言ゲートの実装も OK

### 3. 要対応 2 件の修正

- **`specs/00-identity.md`:** 「わたしの構成」を本元 / プロジェクトの両視点併記に
  変更。本元では `core/`, `skills/`, `specs/` が直下にあり直接編集する、
  プロジェクトでは `.nxt-core/` 配下で読み取り専用、と明確化
- **`specs/03-tools.md`:** 構成ツリーを場所非依存で記載し、「編集ルール」
  セクションで本元（直接編集）とプロジェクト（読み取り専用）を区別
- **`specs/01-workflow.md`:** 「本元リポジトリ向け: システム調整フロー」を新設
  - `/dsgn`, `/tp`, `/codi` は本元対象外（`SOURCE_EXCLUDED_SKILLS` 参照）を明記
  - 4 ステップフロー（対話 → 直接編集 → 動作確認 → /wrap）
  - 4 つの原則（設計書不要・記録は /wrap 集約・品質チェック対象外・giget 配布前提）
  - 本元で使える補助スキル一覧

## SLOG-20260411-005.決定事項 — 決定事項

1. **スコープは作業の種類から機械的に決まる** (`specs/07-scopes.md`)
   - 理由: 視点宣言しても、その後の判断でスコープを裏切る事故が起きる。
     判定を作業種類から機械化することで AI の判断余地を潰す
   - 3 種類: システム調整 / プロジェクト拡張 / プロジェクト実コンテンツ
   - 判断の順序: 作業種類 → スコープ → 実施場所 → 視点またぎ禁止

2. **本元リポジトリでは `/dsgn`, `/tp`, `/codi` を使わない** (`specs/01-workflow.md`)
   - 理由: 実装上すでに `SOURCE_EXCLUDED_SKILLS` で除外されているのに specs が
     明示していなかった。本元の作業はシステム調整であり、
     プロジェクト向けの 3 フェーズ開発フローとは別物
   - 本元フロー: 対話 → 直接編集 → 動作確認 → /wrap

3. **specs は本元 / プロジェクトの両視点で読めるよう併記する**
   (`specs/00-identity.md`, `specs/03-tools.md`)
   - 理由: specs が「プロジェクト視点」一辺倒だと、本元で作業する AI が
     混乱して視点ブレを起こす。今日の失敗の根本原因の一部
   - 具体: `.nxt-core/` は本元には存在しないので、両視点で明示する

4. **09-memory.md の state 移行検討中記述は今回触らない**
   - 理由: 実害が低く、本題（プロジェクト OS 側の /codi レイアウトデザイン工程）
     を邪魔したくない。移行するか削除するかの決定は別セッションで行う
   - 状態: 未決定のまま保留
