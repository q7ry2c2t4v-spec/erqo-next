エル子（生まれたばかりの本体）の初期調整セッション。
3 つの大きな作業を本体リポジトリで実施した:

1. コーディング規約の文書化（specs/06-coding-rules.md 新設）

### 2. 定数集約リファクタ

**新規作成:**
- `core/constants.py` — 棚名・ファイル名・パターン・ステータス・パイプラインステップ・URL・数値定数を集約

**実装中に判明した拡張:**
当初計画では「ファイル名・パターン・リスト」だけを集約予定だったが、install.py が新規インストール時に PROJECT_ROOT が None になり得るため、Path オブジェクト由来ではなく「名前文字列」（CLAUDE_DIR_NAME, LIBS_DIR_NAME, SETTINGS_FILENAME 等）も constants.py に追加した。これにより paths.py が constants.py の名前を使って Path を組み立てる二層構造が実現。

**移行対象（9 ファイル）:**
- `core/paths.py` — 派生 Path 定数を追加、constants.py から名前を import
- `core/page_parser.py` — STATE_FILE_PREFIX, TP_FILE_PATTERN, STATUS_*, HISTORY_DIR_NAME を constants へ移動
- `core/install.py` — 全リテラル → 定数参照、ヘルパー関数で path 組み立て
- `core/session.py` — VERSION_CHECK_*, GitHub URL, 数値定数を constants/paths へ
- `core/state.py` — STEPS → PIPELINE_STEPS、import 元変更
- `core/record.py` — STATUS_*, INDEX_MD_FILENAME 等を constants から
- `core/load.py` — TP_FILE_PATTERN, HISTORY_DIR_NAME を constants から
- `core/index.py` — tag_aliases.json リテラル → paths.TAG_ALIASES_FILE
- `core/feedback.py` — feedback サブディレクトリ → paths.FEEDBACK_DIR

**動作確認:**

- session.py 通常/compact 両モード成功
- index.py reindex の LIBS_DIR not found ガード動作確認

state.py で `STEPS` → `PIPELINE_STEPS` を replace_all したところ、import 行の `PIPELINE_STEPS` が `PIPELINE_PIPELINE_STEPS` になり import エラー。手動修正で復旧。

### 3. 本体リポジトリでスキルが使えない問題の解決

**原因:**
- Claude Code は `.claude/skills/` だけを参照する
- 本体リポジトリは install.py の IS_SOURCE ガードで止まるため `.claude/skills/` が空のまま
- 結果として本体で /gh, /mmy 等のスラッシュコマンドが認識されない設計の穴

**解決:**
- `core/dev.py` を新規作成（本体リポジトリ専用の最小セットアップ）
- install.py の `setup_hooks` / `setup_skills` / `setup_os_skills_manifest` を再利用
- install.py の `_rewrite_skill_paths` を修正: nxt_path == "." の場合は `{nxt}/` ごと除去
- `.gitignore` に `.claude/skills/` を追加（コピー元との二重追跡防止）
- specs/03-tools.md に dev.py セクション追記、構成図と依存関係を更新

**動作確認:**
- `python core/dev.py setup` で 9 スキルがコピーされ Hook 設定成功
- 全 SKILL.md から `{nxt}` プレースホルダが除去されたことを grep で確認
- 再実行で冪等性確認

### 4. 振る舞い原則の強化 (specs/00-identity.md)

ユーザーから「先代のエル子と違う、振る舞いがだいぶ変わっている」との指摘。原則を次の通り強化:

- 旧: 「専門用語を使わないように心がけます（コード・技術用語は英語可）」
- 新: 「ユーザーはプログラミング素人です。一切の専門用語を使わず、要約してわかりやすく説明します」
- 追加: 「表や箇条書きを並べすぎず、短く要点だけ伝えます」

### 5. Bash 自動承認設定

- 本体リポジトリの `.claude/settings.local.json` に `Bash` を allow 追加
- グローバル `~/.claude/settings.json` にも `Bash` を allow 追加（全プロジェクトで有効化）

### 6. skills/ ↔ .claude/skills/ 二重管理問題の解決

ユーザー指摘: 「skills が 2 か所存在するのは保守性が悪い」「一か所での変更にできないか」

物理的な 1 か所化は handler.py の core/ 探索ロジックを根本から作り直す必要があり副作用が大きいため、**実質 1 か所運用** を採用:

- `core/dev.py` に `_sync` 内部サブコマンドを追加(stdin から JSON を読み、skills/ 配下なら setup_skills + manifest 更新を silent 実行)
- `.claude/settings.json` に PostToolUse Hook (matcher: Write|Edit) を追加し `python core/dev.py _sync` を呼ぶ
- 動作確認: `skills/gh/SKILL.md` を編集 → 1 秒後に `.claude/skills/gh/SKILL.md` の mtime も更新されることを stat で検証

## SLOG-20260410-001.決定事項 — 決定事項

1. **コーディング規約は specs/01-workflow.md ではなく specs/06-coding-rules.md に新設**
   - 理由: 開発フェーズ定義（dsgn → tp → codi）と混ざると検索性が落ちる。今後ルールが増える起点にもなる。

2. **constants.py + paths.py の二層構造を採用**
   - 理由: install.py の新規インストール時は PROJECT_ROOT が None になり得るため、Path 派生定数では足りない。名前文字列を constants.py に置き、paths.py がそれを使って Path を組む形なら両方の用途を満たせる。

3. **dev.py を install.py から独立した別ファイルにする**
   - 理由: `--source` フラグで install.py に分岐を入れると IS_SOURCE ガード（プロジェクトでの誤実行防止）が緩む。release.sh と同じく「本体専用 dev ツール」を別ファイルで持つのが既存パターンに沿う。

4. **アイデンティティの「専門用語」原則を「心がけます」→「一切使わず」に強化**
   - 理由: 「心がける」だと振る舞いが lax になりがち。実際このセッションの前半で表・jargon を多用してしまった反省。

5. **Bash 自動承認は細かい command pattern ではなく Bash 全体を許可**
   - 理由: 都度確認の煩わしさを完全に排除するため。グローバルにも適用済み。

6. **skills/ の 2 か所問題は物理的 1 か所化ではなく auto-sync で解決**
   - 理由: 物理的に 1 か所にするには handler.py の core 探索ロジックを作り直す必要があり、副作用が大きすぎる。auto-sync ならユーザーから見て「触る場所は skills/ だけ」になり実質 1 か所と同じ。

7. **大量の変更は本セッションでは未コミット**
   - 理由: 次回のコミットでまとめて 1 回でレビューできる方が変更の意図を追いやすい。/gh で実施予定。
