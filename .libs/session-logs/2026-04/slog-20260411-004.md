# SLOG-20260411-004 — 視点宣言ゲートを session.py に組み込み

関連: SLOG-20260411-003
タグ: セッション開始, 視点宣言, スコープ, 本元OS, プロジェクトOS, Hook, session.py, session-start, orientation-gate, scope, is-source, hook

## SLOG-20260411-004.概要 — 概要

ユーザーから「本来はプロジェクト OS 側の /codi レイアウトデザイン工程を
定義したいが、その前にエル子が毎セッション OS の本筋・本元/プロジェクト
の視点を忘れることが致命的」という問題提起を受けた。対策として
`core/session.py` に視点宣言ゲートを 1 関数追加する最小構成で合意し、
実装・動作確認まで完了した。

## SLOG-20260411-004.作業内容 — 作業内容

### 1. 問題提起の整理

- 症状: エル子が毎セッション冒頭で本元 OS か プロジェクト OS かを
  曖昧にしたまま作業を始めてしまう
- 影響: スコープが曖昧なまま進めると、知らなかったシステム機能間の
  依存を通じて致命的な副作用が発生しうる
- ユーザーの希望: 毎セッション開始時に、システム全体 (specs/ + core/
  + skills/) の構造を理解した上で、現在の視点を必ず宣言してから
  作業に入る
- 重要な切り分け: ここで言う「システム理解」は specs/ + core/ の話
  であって、.libs/ (プロジェクト資料) とは物理的に別物

### 2. 提案の段階的絞り込み

1回目: 案A (本筋サマリ表示) / 案B (4項目チェックリスト) / 案C (両方) を提示
→ ユーザー: 「4項目では不十分。全システム理解と視点が必須」
→ /ret を二部構成化 (Part A = システム自己診断、Part B = セッション復元) する案を提示
→ ユーザー: 「複雑にしたくない。1つだけ推奨して」
→ 最終: session.py に関数 1 つ追加する最小構成に決定

### 3. 実装

- `core/session.py` に `build_orientation_gate()` を追加
  - `IS_SOURCE` の値から「本元 OS」「プロジェクト OS」ラベルを機械的に決定
  - 視点・スコープ・根拠を最初の応答冒頭で宣言するよう指示するブロックを返す
- `build_normal_context()` と `build_compact_context()` の `parts = []` を
  `parts = [build_orientation_gate()]` に変更
- `specs/05-session.md` に「視点宣言ゲート (必須・前置)」セクションを追加

### 4. 動作確認

- `python core/session.py` (通常モード): 先頭に視点宣言ゲート → 通常 7 項目が続く
- `python core/session.py --compact` (compact モード): 先頭に同ゲート → compact 3 項目が続く
- 本元リポジトリで実行したので `IS_SOURCE=True` → 「本元 OS」と表示された
- プロジェクト側では自動的に「プロジェクト OS (IS_SOURCE=False)」に切り替わる

## SLOG-20260411-004.決定事項 — 決定事項

1. **視点宣言ゲートは session.py に組み込む** (`/ret` でも新スキルでもない)
   - 理由: SessionStart Hook で自動発動するため取りこぼしがない。
     既存 Hook に 1 関数追加するだけで複雑さが最小
   - 場所: `core/session.py` の `build_orientation_gate()`

2. **IS_SOURCE の機械判定で視点を提示する**
   - 理由: エル子の判断ではなくファイルシステム状態から決まるため
     取り違えが起きない (`specs/08-responsibility.md` の機械的判定と整合)
   - 表示: 「本元 OS (IS_SOURCE=True)」「プロジェクト OS (IS_SOURCE=False)」

3. **エル子は視点・スコープ・根拠を宣言するまで作業に入らない**
   - 理由: 受動的に specs を読むだけでは取り違えが起きる。能動的に
     書き出して初めて照合が成立する
   - 宣言項目: 視点 / 依頼スコープ (本元・プロジェクト・共通) / 根拠

4. **システム spec と プロジェクト資料を物理的に切り分ける**
   - 理由: .libs/ の過去資料を混ぜると「システム理解」と「前回作業」が
     一緒くたになり、視点確認が薄まる
   - ゲートは specs/ + IS_SOURCE だけを参照。`.libs/` は触らない

5. **通常モード・compact モード両方に前置する**
   - 理由: コンパクション後も視点を保持する必要がある
   - 実装: `build_orientation_gate()` を両関数の parts 先頭に入れる
