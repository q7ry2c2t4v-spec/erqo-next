## SLOG-20260411-007.作業内容 — 作業内容

### 1. 判定機械化フェーズ (core/load.py)
`_score_keywords` 共通関数を新設し、`_judge_ui` と新設 `_judge_layout` が同じスコアリング方式 (ID +3 / タグ +2 / 説明 +1、閾値 2) を共有する構造に DRY 化。`load_task` で layout/ui/all の 3 値化 + 包含展開 (layout → `task_types=["layout","ui","all"]`)。`_collect_rules` を `task_type: str` → `task_types: list[str]` に変更し、マッチ条件を `scope == "all" or any(tt in scope for tt in task_types)` に。`UI_ID_PATTERNS` / `UI_TAG_KEYWORDS` を constants.py に移動。4 仮タスク (LAYOUT-HOME-CLONE / UI-CARD-01 / AUTH-LOGIN-API / PAGE-ABOUT-CLONE) で動作確認 OK。

### 2. clone.py 最小骨格 (新規)
7 サブコマンド (`recon` / `dump` / `apply` / `rules` / `build` / `assemble` / `baseline`) の dispatch を持つエントリポイント。`SUBCOMMANDS = {name: globals()[f"cmd_{name}"] for name in LAYOUT_SUBSTEPS}` で動的生成し、サブステップ名の二重管理を排除 + cmd_* 関数が欠ければ import 時に KeyError で欠落検知。各 `cmd_*` は `_stub()` 経由で「未実装」+ exit 0。IS_SOURCE ガードなし (本元でも起動可能)。動作確認 OK (7 サブコマンド全通 + 引数エラー処理)。

### 3. recon A (入力ファイル規約)
「本棚ページが正本」原則に基づき `.libs/storybook/<task-slug>/input.md` を参考 URL 入力源に採用。`_input_path` / `_build_input_template` / `_extract_reference_urls` を新設し、`parse_sections` で `## <TASK-ID>.参考サイト` セクションを検出、`-` / `*` 両形式の箇条書きから URL を抽出。constants.py に `STORYBOOK_SHELF_NAME` / `STORYBOOK_INPUT_FILENAME` / `RECON_REFERENCE_SECTION_SUFFIX` / `RECON_REQUEST_SECTION_SUFFIX` / `RECON_NOTE_SECTION_SUFFIX` を追加 (後者 2 つは apply/dump 用の前置定義)。`LIBS_SHELVES` が `STORYBOOK_SHELF_NAME` を参照する形に修正して二重管理解消。3 ケース動作確認 OK。

### 4. recon B (Playwright スクショ最小動作)
当初は `npx playwright --version` で検査する設計だったが、本元で実行したところ **npx が未インストール時に on-demand でパッケージを自動取得 (playwright@1.59.1) してしまう**ことを実測で発見。対策として `PROJECT_ROOT/node_modules/.bin/playwright` を `shutil.which("playwright", path=str(bin_dir))` で直接探す方式に変更 (Windows の `.CMD` / `.EXE` もクロスプラットフォーム解決)。`_find_playwright_cli` は `(path, exit_code, msg)` を返し、exit 3 (node_modules なし) / exit 4 (playwright なし) / exit 0 を明示分岐。`_run_playwright_screenshot` は `subprocess.run([playwright_path, "screenshot", ...])` を timeout 60 秒で呼ぶ。本元では実スクショ確認は行わず、3 ケース (node_modules なし / tmpdir モックで playwright なし / ダミー .CMD 配置) で構造検証 OK。

### 5. recon C 縮小版 — 設計方針の議論を経て着地
当初「Node スクリプト (`core/clone_recon.mjs`) を新設して 6 項目取材 + mobile まで一気に進める」案を提示。ユーザーから「あえて node を使わない方針でいた」「過去にトラウマ」との指摘を受け、調査したところ `core/` の 16 モジュール全てが Python 標準ライブラリのみで動作し、`requirements.txt` / `pyproject.toml` も存在しない (pip 依存ゼロ) ことが判明。3 案 (CLI 縮小 / Node 追加 / playwright-python 追加) を比較し、**playwright-python も標準ライブラリのみ方針を破るため Node より重い決断**であることを認識。ユーザー判断で案 II (CLI 縮小) を採用。`RECON_VIEWPORTS = [("desktop", (1440,900), ...), ("mobile", (390,844), ...)]` をリスト化し、`cmd_recon` を viewports ループ構造に変更。tmpdir + モックで desktop → mobile の 2 回呼び出しを確認 OK。

## SLOG-20260411-007.決定事項 — 決定事項

1. **task_type の 3 値化 + 包含展開** — layout タスクには `task_types=["layout","ui","all"]` を渡し、`_collect_rules` が scope マッチで全段適用する。理由: レイアウトは UI のサブタイプ (layout ⊂ ui) なので UI 向けルールも自動適用したい
2. **SUBCOMMANDS は LAYOUT_SUBSTEPS から動的生成** — `globals()[f"cmd_{name}"]` で生成し、二重管理回避 + 欠落検知を兼ねる。理由: constants 単一出どころ原則 (specs/06) の徹底
3. **clone.py は IS_SOURCE ガードなし** — 本元でも起動可能。理由: SLOG-20260411-006 の 8 段階計画 7 番「本元では /codi が使えないので clone.py 単体で起動する形」に沿う
4. **URL 入力方式は `.libs/storybook/<task-slug>/input.md`** — 参考サイト URL + 要望 + メモを 1 ファイルに集約。理由: 「本棚ページが正本」原則 (SLOG-20260411-006 決定事項 3)、apply (要望適用) の入力もここから取れる
5. **Playwright 検査は `node_modules/.bin/playwright` を直接探索** — `npx playwright --version` は npx の on-demand install で検査にならないことが実測で判明したため。`shutil.which(path=...)` で Windows `.CMD` を自動解決
6. **本元での実スクショ確認は不実施** — Node + chromium を本元に入れるのは specs/01-workflow.md の「本元は Python + Markdown 中心」に反する。本元ではエラーケース + 構造検証のみ、実動作確認は別環境に委ねる
7. **recon C を CLI 範囲に縮小** — `core/` の 16 モジュール全てが Python 標準ライブラリのみで動く暗黙ルール (pip 依存ゼロ、requirements.txt なし) が判明。Node 追加も playwright-python 追加もこの暗黙ルールを破る重い決断であり、今の段階では必要最小限 (2 枚スクショ) で止めて recon D 着手時に方針を再検討する。判断先送りで ROI を最大化
8. **新設定数は単一出どころ原則を厳守** — `LIBS_SHELVES` が `STORYBOOK_SHELF_NAME` を参照、`RECON_VIEWPORTS` が viewport タプルとファイル名定数を参照する構造で二重管理排除
