## SLOG-20260411-001.作業内容 — 作業内容

### 1. 本元用スキル除外機能の追加

本元では `/dsgn`, `/tp`, `/codi` は実質使われないのに Claude Code のスキル一覧に
表示されていた。これを共通スコープの設計パターンで除外できるようにした。

- `core/constants.py` に `SOURCE_EXCLUDED_SKILLS = frozenset({"dsgn", "tp", "codi"})` を追加
- `core/install.py` の `_scan_skills()` で `IS_SOURCE == True` のときのみフィルタ
- `python core/dev.py update` を実行、stale 削除機能が本元 `.claude/skills/` から 3 つを自動削除
- 結果: 本元では 6 スキル (aud, fb, ret, rsrc, ruru, wrap) のみ表示
- プロジェクト側には引き続き全スキル配信 (IS_SOURCE == False なので影響なし)

### 2. ルール A — デフォルトスコープは共通

ユーザー指示:「機能追加・修正の際、何も言わなければ共通処理、本元/プロジェクト
だけの場合はちゃんと言う」。`specs/07-scopes.md` の末尾「違反時の対応」の直前に
「## デフォルトスコープ」セクションを追加:

- ユーザー指定がなければ共通スコープを前提
- 本元/プロジェクト限定はユーザー明示のときのみ
- 迷ったら確認 (曖昧判断禁止・自己判定禁止と整合)
- 共通実装は「constants.py に出どころ + install.py/dev.py で配置」パターン

### 3. ルール B — OS 記録機能を最優先

ユーザー指示:「Claude Code のメモリー機能はデバイス依存なので OS 理念に反する。
OS の記録機能を最優先」。`specs/09-memory.md` を新設:

- 3 つの記憶層を整理 (OS 記録 / Claude Code メモリー / 会話履歴)
- マルチデバイス共有の可否で優先順位を決定
- OS 記録 (`.libs/`, `.claude/state/`, `specs/`) が正本
- Claude Code メモリーは Claude Code 自身の操作 tips 程度に限定
- 同じ情報を両方に持つ場合は OS 記録が正本、Claude Code 側は複製扱い
- ついでに Claude Code の保護パス知識 (`.git/` と `.claude/`、例外 4 つ) を常設化

`CLAUDE.md` に `@specs/09-memory.md` を追記。

### 4. Claude Code メモリーの整理

既存の 2 件を specs に統合し、Claude Code メモリー側を削除:

- `feedback_no_self_judge.md` → 内容は `specs/08-responsibility.md` に既存、削除
- `feedback_claude_protected_paths.md` → 内容を `specs/09-memory.md` に統合、削除
- `MEMORY.md` を書き換え: 「OS 側を正本とする、Claude Code メモリーは OS 無関係な
  Claude Code 自身の tips のみ使用」と明記

### 5. 先日の共通 permission 修正の確認

ユーザー質問:「先日のコマンドスキップ修正は共通用に修正しましたか？」
`core/install.py`, `core/dev.py`, `core/constants.py` を grep して、SLOG-004 の
共通 permission 設定 (`setup_permission_defaults`) が「constants に定数 +
install に関数 + dev から再利用」の鉄則通りに実装済みであることを確認。問題なし。

## SLOG-20260411-001.決定事項 — 決定事項

1. **本元では dsgn/tp/codi を非表示にする**
   - 理由: 本元は既に `specs/` で設計済み、これらのスキルは実質使われない
   - 実装: `SOURCE_EXCLUDED_SKILLS` (constants.py) + `_scan_skills()` に IS_SOURCE フィルタ
   - 除外リストは将来変更可能 (定数 1 か所で管理)

2. **機能追加・修正のデフォルトスコープは共通** (ルール A)
   - 理由: ユーザーは基本的に本元・プロジェクト両方で動く形を望む
   - 本元/プロジェクト限定は明示指定のときのみ
   - 保存先: `specs/07-scopes.md`

3. **OS 記録が正本、Claude Code メモリーは補助のみ** (ルール B)
   - 理由: Claude Code メモリーはデバイス依存で複数デバイス共有できず、OS 理念に反する
   - OS 記録 (`.libs/`, `.claude/state/`, `specs/`) が正本
   - 保存先: `specs/09-memory.md` (新設)

4. **OS 全体ルールは `specs/`、プロジェクト固有ルールは `/ruru` で `.libs/rules/`**
   - 理由: 両方に置くと重複・片方の直し忘れが発生する
   - 今回のルール A/B は OS 全体ルールなので `/ruru` は使わず `specs/` に配置

5. **Claude Code の保護パス知識を OS 側に常設化**
   - 理由: 会話ごとに情報を拾い直すのではなく、毎セッションで自動読み込みされる場所に置く
   - `specs/09-memory.md` 内にルール化 (`.git/` と `.claude/`、例外 4 つ)
