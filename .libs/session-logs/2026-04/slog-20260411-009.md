## SLOG-20260411-009.決定事項 — 決定事項

1. **リサーチ構成は「1 言語/FW につき ARCH + QUALITY の 2 ページ」固定** — 今後の言語/FW 追加もこの形式。理由: 観点 1-3 (構成・境界・データ) と 4-6 (品質・テスト・本番化) を分離して読み直しやすく
2. **specs/06-coding-rules.md は肥大化禁止** — 1 ファイル 1 テーマ原則を保つため、L2/L3 は `specs/coding/` サブディレクトリで分割。L1 だけハブに残す。理由: CLAUDE.md 経由で常時コンテキストに載るのは L1 の 124 行のみに抑え、L2/L3 は必要時に Read
3. **サブディレクトリ名は `specs/coding/`** (`rules/` は紛らわしいと却下) — 「コーディング規約」が一目で伝わる
4. **層レベルはファイル名 prefix で示す** (`l2-` / `l3-`) — 機械判定スクリプトが正規表現で扱いやすく、新カテゴリ追加時もファイル 1 つ足すだけ
5. **適用判定は `core/coding_rules.py` が機械的に行う** — AI の自己判断ゼロ (`specs/08-responsibility.md` 整合)。ファイル拡張子 + プロジェクトルートのマーカーファイル + パス部の `workers/` の組合せで機械判定
6. **コーディングルールの注入は PreToolUse Hook 自動化 (Option B 採用)** — Option A (specs 追記のみ) では呼び忘れが残るため。Hook は情報提供のみで `exit 0`、ブロックしない (連続編集の煩わしさを避ける)
7. **Hook は `write_guard.py` と共存** — 既存 `setup_hooks()` の matcher 重複弾きを「matcher + コマンド単位マージ」に拡張。同 matcher 内の `hooks` 配列に複数コマンドを並べる構造で、順序は `write_guard.py` (ブロック) → `coding_rules_hook.py` (情報提供)
8. **本元 `core/` 直下は引き続き Python 標準ライブラリのみ** — `l2-python.md` §1 で本元/プロジェクト棲み分けを明文化。uv/Ruff/pyright/pytest はプロジェクト側のみ適用
9. **Cloudflare Workers ルールはプロジェクト固有マーカーで判定** — `wrangler.jsonc|json|toml` のいずれかが project_root に存在 or 対象ファイルパスに `workers/` が含まれる、いずれかで L3-Cloudflare 適用
