設計中にユーザーから「AI の自己判定禁止」の強い指摘があり、OS 全体ルールとして

### 3. 責任分担ルール制定 (本セッションの核)

ユーザー指摘: 「エル子が自己判定します」は曖昧判断禁止・工程省略禁止に違反。
調査の結果、`skills/codi/SKILL.md:171` に /codi 限定で部分的にだけ書かれており、
OS 全体のルールとして未定義だった。

- `specs/08-responsibility.md` 新設:
  - 3 つの禁止事項 (工程省略 / 曖昧判断 / 発動条件の自己判定)
  - 役割分担表 (python スクリプト vs AI)
  - 新機能設計時の自問チェックリスト
  - AI 記憶への依存禁止 (情報源は git diff / スクリプト出力 / state.py / ファイルシステムに限定)
- `CLAUDE.md` に `@specs/08-responsibility.md` を追記
- `skills/aud/SKILL.md` の違反箇所を修正:
  - 手順1: 「会話履歴から拾う」→ `git_ops.py check` 出力のみ
  - 手順2: 「会話中で削除・リネームしたシンボル抽出」→ `git diff` 出力から機械抽出
  - 手順3, 4: 実行可能コードのみに発動条件明記
  - 手順5: 参照 spec を機械的に決定するテーブル追加
- メモリに feedback を永続保存

### 4. features TEMPLATE に「参考リサーチ」セクション追加

当初は新サブコマンド新設 + 自己判定による発動の案を考えたが、ユーザー指摘
「既存 record.py に 1 セクション追加するだけでは？」が完全に正しかった:

- `core/record.py` の TEMPLATE を 6 → 7 セクション (「トラブル」と「備考」の間に
  「参考リサーチ」挿入)
- `skills/codi/SKILL.md` ステップ4a の説明文を 7 セクション版に更新
- 発動条件・テンプレ生成は record.py が制御、AI は中身を書くだけ
- リサーチがなければ「該当なし」と書く (既存「トラブル」セクションと同じ扱い)
