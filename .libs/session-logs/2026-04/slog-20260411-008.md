## SLOG-20260411-008.決定事項 — 決定事項

1. **取材方式は甲 (Node + Playwright)** — 本元の「Python 標準ライブラリのみ」原則は **`core/` 直下を Python のみに保つ** 形で維持し、Node スクリプトは新設の `core/clone_node/` サブディレクトリに置くことで両立。配布は giget で自動的に届く
2. **複数 URL は site-<i> 分割** — input.md に複数 URL を書くと recon が `site-1` / `site-2` ... のサブディレクトリで分けて取材し、dump で 1 ページに統合する。理由: 「途中で別サイトを追加」要件を冪等に満たすため
3. **URL なしモード = `text-only.json` マーカー** — URL ゼロ + 要望ありで recon が `text-only.json` を書いて成功扱い。dump 以降がそのマーカーを見て「要望だけから本棚ページを構築」モードに入る。理由: 「テキストだけでデザイン開発」要件
4. **冪等性の徹底** — apply / rules / build / assemble / baseline は `_replace_or_append_section` でセクション単位冪等置換。要望や取材結果が変わったら再実行で本棚ページ全体が最新化される (途中修正対応)
5. **責務分離: スクリプトは指示書、AI は実コード** — build / assemble / baseline はスクリプトとして「指示 + 雛形 + 生成先パス」を本棚ページに追記するだけで、`.tsx` / `page.tsx` の実コード生成は AI に委ねる (specs/08-responsibility.md 「python はタスク管理、AI は内容執筆」原則)
6. **state.py のレイアウト判定は ID パターンのみ** — load.py の `_judge_layout` はタグも参照するが、state.py は本元でも動く必要があるので LAYOUT_ID_PATTERNS のみで判定。LAYOUT/PAGE/CLONE/DESIGN 規約での運用が前提
7. **本棚ページが全工程の正本** — dump 以降の全サブステップが `.libs/storybook/<slug>/<slug>.md` を読み書きし、`.tsx` / `.stories.tsx` の直接編集は禁止。修正は必ず input.md → recon/apply の経路
8. **本元では実取材しない** — Node + chromium の本元インストールは specs/01-workflow.md (本元は Python + Markdown 中心) と相反するため、実 URL 取材はプロジェクト側に委ねる。本元は構造検証のみ
