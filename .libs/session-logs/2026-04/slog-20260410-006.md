# SLOG-20260410-006 — permission 保護パス調査と equf4 知見取得

関連: SLOG-20260410-005
タグ: permission, 保護パス, bypassPermissions, 一時ファイル, equf4, Hook, protection, workaround

## SLOG-20260410-006.概要 — 概要

SLOG-005 で /wrap 実行中に発生した permission 確認プロンプトの原因を掘り下げ、
Claude Code の保護パス仕様 (`.git/` と `.claude/` 配下) を特定。一時ファイルの
置き場所を 2 回移動して完全回避に至った。並行して過去プロジェクト `equf4` を
調査し、erqo-next に取り入れたい自動化 Hook のアイデアを複数取得した。

## SLOG-20260410-006.作業内容 — 作業内容

### 1. 1 回目の permission 問題と修正

/wrap 実行中、`.git/COMMIT_MSG_TMP.txt` への Write/rm で 2 回プロンプトが出た。
原因: `.git/` 配下は Claude Code が sensitive file 扱いで bypassPermissions の
例外。対応: 一時ファイルを `.claude/state/commit_msg_tmp.txt` に移動 (コミット 8b5f11d)。

### 2. 2 回目の permission 問題 (VS Code diff プロンプト)

移動先の `.claude/state/commit_msg_tmp.txt` でも VS Code の diff 確認プロンプトが
出た。claude-code-guide エージェントで調査した結果:

- `.claude/` 配下も保護パスで、`bypassPermissions` の例外
- 例外サブディレクトリは `commands/`, `agents/`, `skills/`, `worktrees/` の 4 つだけ
- `--dangerously-skip-permissions` CLI フラグも保護パスは例外
- 環境変数・settings.json による一括無効化は存在しない

対応: 一時ファイルをプロジェクトルート直下 `.commit_msg_tmp.txt` に最終移動
(コミット 9df2582)。以降プロンプトなしで dogfooding 成功。

### 3. equf4 プロジェクト調査

ユーザー提供の参照元 `C:\Users\prima\Desktop\devs\equt` (equf4、erqo の祖先 OS) を
Explore エージェントで thoroughly 調査。発見:

- equf4 は「確認を無効化」ではなく「確認が不要な設計」で全自動実装を実現
- 一時ファイル・ランタイムデータは `state/` (プロジェクトルート直下、保護パス外)
- `.claude/hooks/` に 5 つの MJS スクリプト (write-guard, bash-guard, transition,
  completion, stage-checkpoint)
- state マシン (DESIGNING/IDLE/PLANNING/IMPLEMENTING/COMPLETING) で合法遷移のみ Hook で許可
- AI の判断を Hook が物理的に代行することで確認プロンプトを発生させない設計

### 4. 将来の検討候補を整理

equf4 から転用したいアイデアを優先度付きで整理し、今セッションでは実装せず
次回以降に回す判断をした (未完了セクション参照)。

## SLOG-20260410-006.決定事項 — 決定事項

1. **一時ファイルは必ずプロジェクトルート直下に置く**
   - 理由: Claude Code の保護パス `.git/` と `.claude/` (例外 4 つを除く) への
     書き込みは `bypassPermissions` でも確認プロンプトが出る。設定で無効化する方法は存在しない
   - 適用: `/wrap` の `.commit_msg_tmp.txt` が第 1 号。今後の一時ファイルも同様

2. **Claude Code の保護パス仕様をメモリに永続化**
   - 理由: 今セッション中に 2 度同じ罠にハマった。知識として永続化しないと同じ失敗を繰り返す
   - 保存先: `feedback_claude_protected_paths.md`

3. **equf4 の Hook 設計思想を将来の erqo-next に取り入れる**
   - 理由: 「確認を無効化」ではなく「設計で回避」が本質。Hook で物理的に制御する
     equf4 の思想は erqo-next の責任分担ルール (specs/08-responsibility.md) と一致する
   - 今セッションでは実装せず、未完了に記録して次回以降に実装

4. **permission 問題の修正は fix: コミット 2 つとして履歴に残す**
   - 理由: 経緯を git 履歴上で追えるようにするため (8b5f11d, 9df2582)
