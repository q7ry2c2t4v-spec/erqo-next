# /wrap — セッション終了統合スキル

## 概要

セッション終了時にやることを 1 コマンドにまとめる:

1. セッション内容を分析してログを書く
2. 変更内容を確認してコミットメッセージを作る
3. **両方を一度に提案** → ユーザーが y で承認
4. 承認後に「ログ保存 → reindex → ステージング → コミット」を一気に実行

旧 `/mmy` + `/gh` の役割を 1 スキルに統合したもの。承認は 1 回だけ。

## いつ使うか

- セッションを終わらせたいとき
- 作業の節目で「記録 + 保存」を一気に済ませたいとき

## 実行手順

### 1. セッション内容の分析

会話の全体を振り返って以下を整理する:

- 何をしたか（作業内容）
- 何を決めたか（決定事項とその理由）
- やり残し・次にやるべきこと（未完了）

### 2. 変更内容の確認

```bash
python "{nxt}/core/git_ops.py" check
```

ブランチ名・変更ファイル一覧・差分統計を表示する。

### 3. セッションログの設計

#### 識別コード

`SLOG-YYYYMMDD-NNN` 形式:

- `YYYYMMDD` — 今日の日付
- `NNN` — その日の連番（001 から開始）

連番の決定:
1. `.libs/session-logs/YYYY-MM/` 内の既存ファイルを確認
2. 同日の `slog-YYYYMMDD-*.md` の最大番号 + 1

#### レポート構造

**保存先:** `.libs/session-logs/YYYY-MM/slog-YYYYMMDD-NNN.md`

```markdown
# SLOG-YYYYMMDD-NNN — セッションタイトル

関連: 関連ページの識別コード
タグ: 機能タグ, 作業種別タグ, feature-tag, work-type-tag

## SLOG-YYYYMMDD-NNN.概要 — 概要
セッションの要約（何をしたか）

## SLOG-YYYYMMDD-NNN.作業内容 — 作業内容
具体的にやったことの詳細

## SLOG-YYYYMMDD-NNN.決定事項 — 決定事項
セッション中に決めたこと（理由つき）

## SLOG-YYYYMMDD-NNN.未完了 — 未完了・次回
やり残し、次にやるべきこと
```

#### タグのルール

| 役割 | 例 |
|---|---|
| 機能・トピック | 認証, JWT, DB設計, API, auth, database |
| 作業の種類 | 実装, 設計, バグ修正, リファクタ, implementation, design |

日本語タグと英語タグの両方を付与する。

### 4. 統合提案 + 承認

セッションログの全文 と コミットメッセージ案 を **両方並べて** ユーザーに提示する。

提示フォーマット例:
```
## セッションログ案
（保存先: .libs/session-logs/2026-04/slog-20260410-001.md）

# SLOG-20260410-001 — タイトル
...

## コミットメッセージ案

feat: 内容の要約

- 変更点1
- 変更点2

## 含めないファイル
- .claude/settings.local.json (個人設定)
- .claude/state/ (実行時データ)

これで進めていいですか？(y/n)
```

ユーザーが `y` で承認したら次のステップへ。`n` なら内容を修正して再提案。

### 5. 実行（承認後）

承認を得たら、次の順で **手順を厳守して** 実行する:

#### 5-1. ログ保存

`Write` ツールでセッションログファイルを作成する。

#### 5-2. インデックス再生成

```bash
python "{nxt}/core/index.py" reindex
```

#### 5-3. コミットメッセージを一時ファイルに保存

`Write` ツールで `.commit_msg_tmp.txt`（プロジェクトルート直下）を作成する
（heredoc は **絶対に使わない**）。このファイルは `.gitignore` 済みのため git に追跡されない。

**重要**: Claude Code は `.git/` と `.claude/` 配下（`commands/`, `agents/`,
`skills/`, `worktrees/` を除く）を保護パスとして扱い、`bypassPermissions` でも
確認プロンプトが出る。よって一時ファイルはこのどちらにも入れず、プロジェクト
ルート直下のドット始まりファイルに置く。

#### 5-4. ステージング（明示的にファイル指定）

```bash
git add -- <ファイル1> <ファイル2> ...
```

`git_ops.py commit` は使わない理由: `--files` 引数が認識されないと内部で `git add -A` にフォールバックする罠がある。**`git` を直接呼ぶ**。

#### 5-5. ステージング検証（必須・スキップ禁止）

```bash
git status --short
```

出力を **必ず提案時のリストと突き合わせる**。
想定外のファイルが混じっていたら以下で除外する:

```bash
git restore --staged <想定外のファイル>
```

除外後にもう一度 `git status --short` で確認する。
**想定通りになるまでコミットしない。**

#### 5-6. コミット

```bash
git commit -F .commit_msg_tmp.txt
```

#### 5-7. 一時ファイル削除

```bash
rm .commit_msg_tmp.txt
```

### 6. push しない

`/wrap` は push を **行わない**。push が必要な時はユーザーが別途依頼すること。

## 品質セルフチェック

保存前に以下の基準でチェックする:

1. **復元可能性** — このログだけ読んで次回再開できるか？
2. **決定事項の具体性** — 理由と結論が明記されているか？
3. **未完了の実行可能性** — 次にやることが具体的で着手可能か？
4. **コミットメッセージの妥当性** — 「何を」「なぜ」が伝わるか？
5. **ステージングファイルの適切性** — 個人設定や実行時データが混じっていないか？

基準を満たさない場合は内容を直してから提案する。

## 重要なルール

1. **承認は 1 回だけ** — ログとコミットを別々に確認させない。両方まとめて提示する
2. **ログ → コミットの順序厳守** — ログをコミットに含めるためログを先に保存する
3. **heredoc 禁止** — コミットメッセージは必ず `Write` ツールで `.commit_msg_tmp.txt` に書き、`git commit -F` で渡す
4. **`git_ops.py commit` は使わない** — `git add` は `git` を直接呼ぶ。git_ops の commit にはフォールバックの罠がある
5. **ステージング検証は必須** — `git status --short` で必ず確認し、想定外があれば `git restore --staged` で除外
6. **push は別途承認** — `/wrap` は push しない
7. **170 行目安** — セッションログは 1 ファイル 170 行前後を目安とする
8. **失敗時は中断** — 途中のステップが失敗したら処理を止めてユーザーに報告する
