# /aud — 作業監査スキル

## 概要

セッション中の任意のタイミングで、それまでに行った作業を見直すスキル。
古いコードの残骸、依存関係の食い違い、重複、仕様との齟齬を検出してレポートする。

`/ret` と同じ「読むだけ」系。分析して所見を提示し、修正はユーザー承認を取ってから実行する。

## いつ使うか

- 作業の節目で「これまでの変更に取りこぼしがないか確認したい」とき
- `/wrap` でセッションを閉じる直前のセーフティネットとして
- 大きめのリファクタ・削除作業の後に依存関係の追従漏れをチェックしたいとき
- 「古いコードの残骸、消し忘れはないか？」と不安になったとき

## 実行手順

### 1. 変更対象の特定

```bash
python "{nxt}/core/git_ops.py" check
```

監査対象は **`git_ops.py check` の出力に含まれる変更ファイルのみ**。
会話履歴や記憶から対象を決めない（`specs/08-responsibility.md` の AI 記憶依存禁止ルール）。

- 対象ファイル 0 件 → 「監査対象なし」と報告して終了
- 対象ファイルあり → 次のステップへ

**運用ルール**: `/aud` はコミット前（未コミット変更がある状態）で呼ぶ。
既にコミット済みの変更は `git status` に現れないため監査対象にならない。

### 2. 依存追跡

対象ファイルごとに `git diff` を実行し、**diff 出力から機械的に**
削除されたシンボル・変更点を抽出する:

```bash
git diff -- <対象ファイル>
```

diff の `-` 行から以下を抽出する（会話の記憶からではなく diff 出力から抽出）:

- **削除された定義** — `-def ...` / `-function ...` / `-class ...` /
  `-<定数名> = ...` / `-import ...` / `-from ... import ...`
- **変更された関数シグネチャ** — `-def foo(x):` と `+def foo(x, y):`
  のように対になる行
- **削除された `@import` / パス参照** — `-@specs/xxx.md` / `-[link](path)`

抽出したシンボルに対して以下を実行する:

1. **残参照チェック** — Grep で全体スキャン、削除シンボルの参照が残っていないか確認
2. **シグネチャ変更の追従** — 変更された関数の呼び出し元を Grep で追跡
3. **参照パス切れ** — 削除/移動したパスを参照している箇所を Grep

情報源は `git diff` の出力のみ。AI 記憶は使わない。

### 3. 古いコード残骸チェック

対象ファイルの拡張子が **実行可能コード** (`.py`, `.ts`, `.tsx`, `.js`,
`.jsx`, `.go`, `.rs`, `.java` 等) の場合のみ実行する。
Markdown・JSON・YAML は対象外。

- **未使用 import** — 変更ファイルで import したが使っていない
- **未使用関数・定数** — 定義したが全体から参照されない
- **消し忘れコメントアウト** — `#` `//` で一時的に消した残り
- **到達不能コード** — `if False:` / 早期 return 後の処理等

### 4. 重複・親和性チェック

対象ファイルの拡張子が **実行可能コード** の場合のみ実行する（手順 3 と同じ基準）。

- **コード重複** — 3 行以上同じ処理が 2 箇所以上
- **ハードコーディング** — `specs/06-coding-rules.md` に照らして
  `constants.py` / `paths.py` に移せるリテラルを検出
- **類似パターン** — 同じ役割の関数が別名で存在する等

### 5. 仕様との整合性チェック

対象ファイルのパスから **参照する spec を機械的に決定** する
（AI が「関連する spec」を判断しない）:

| 対象ファイルのパス | 参照する spec |
|---|---|
| `core/` 配下 | `specs/03-tools.md` |
| `skills/<name>/` 配下 | `specs/02-skills.md` + `skills/<name>/SKILL.md` |
| `specs/` 配下自身 | 変更ファイル自身 + Grep で見つかる参照元 specs |
| `.libs/` 配下 | `specs/04-project-guide.md` |
| `scripts/` 配下 | `specs/03-tools.md` |
| `CLAUDE.md` | `specs/` 配下全て |
| 上記以外 | 対象外 |

各参照 spec の関連セクションが現コードと一致しているか確認する:

- 関数名・コマンド名・引数・手順の一致
- 責任分担・ルール記述との整合（特に `specs/08-responsibility.md`）
- 削除/変更された要素が spec 側にも反映されているか

### 6. レポート提示

所見を 3 段階で構造化して提示する:

```
## /aud レポート

### 変更対象
- 変更ファイル1
- 変更ファイル2

### 問題なし
- 依存関係
- 仕様整合性

### 注意 (動作に影響なし / 見栄え・重複等)
- core/install.py:80 _hook_config で nxt_path == "." の時 /./ が残る
  影響度: 見栄えのみ（paths.resolve() で正規化される）

### 要対応 (動作に影響あり / 整合性破壊)
- 該当なし
```

- **問題なし**: 何もする必要なし
- **注意**: 動作に影響なし。見栄え・重複・将来リスク等。修正は任意
- **要対応**: 動作に影響あり。整合性破壊・未参照残骸・呼び出し元破損等

「注意」「要対応」に修正案がある場合は、**ユーザー承認** を取ってから修正を実行する。

## 重要なルール

1. **読む → 提案 → 承認** — `/aud` 自体は分析だけ。修正は必ずユーザー承認後に実行する
2. **セッション範囲のみ** — プロジェクト全体は見ない。今回触った箇所周辺に絞る
3. **軽量・高速** — 所要時間は短く、全件スキャンや重い静的解析はしない
4. **問題なし報告も必要** — 問題がなければ「問題なし」と明示する（沈黙しない）
5. **影響度を分類** — 「注意」と「要対応」を分けて緊急性を区別する
6. **修正前に依存関係を確認** — 修正を実行する時は、修正箇所の依存先・呼び出し元を確認してから着手する
