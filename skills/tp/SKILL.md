# /tp - タスク計画スキル

## 概要

`/dsgn` で決めたセクションに取り掛かる時に使う。セクション単位で詳細設計とタスク分解を行い、各タスクに識別コードを付与する。

同じセクションに何度でも実行でき、新しい情報（`/rsrc` のリサーチ結果など）を取り込んで計画を更新できる。

## 実行手順

### 1. 対象セクションの確認 + モード判定

`$ARGUMENTS` で指定されたセクションについて:

1. `.libs/design/project-spec/` の設計書を読み、対象セクションの概要を確認する
2. `.libs/design/<section>/` に既存の TP ページがあるか確認する
3. 既存の関連ページを index.py collect で収集する

```bash
python "{nxt}/core/index.py" collect $ARGUMENTS
```

**既存 TP ページがない場合 → 新規モード（ステップ2へ）**
**既存 TP ページがある場合 → 更新モード（ステップ1B へ）**

### 1B. 差分チェック（更新モードのみ）

既存の TP ページと、新たに収集したコンテキストを突き合わせる。

**チェック項目:**

1. **新規情報の特定** — collect で取得したページのうち、既存 TP ページの作成時には存在しなかったものを洗い出す（特に `/rsrc` のリサーチページ）
2. **設計への影響判定** — 新規情報が既存の設計方針・技術選定・タスク分解に影響するかを判定する
3. **矛盾の検出** — 新しい情報と既存 TP の記述内容に矛盾がないか確認する
4. **タスクの過不足** — 新しい情報により、タスクの追加・分割・統合・削除が必要かを判定する

**差分チェックの結果をユーザーに報告する。**

### 2. ユーザーとの対話

セクションの詳細をユーザーと対話しながら詰める:
- 具体的な機能要件
- 技術的な設計方針
- 依存関係・制約事項

**更新モードの場合:**
- 差分チェックの結果を起点に対話する
- どの変更を採用するかユーザーと決定する

### 3. タスク計画の作成・更新

方針が固まったら、承認を取ってからタスク計画を作成・更新する。

各タスクに **カテゴリ-トピック-内容** 形式の識別コードを付与する:
- 英大文字 (`A-Z`)、数字 (`0-9`)、ハイフン (`-`) で構成
- コードだけで「何の資料か」が特定できる説明性を保つ

**保存先:** `.libs/design/<section>/`

```markdown
# TP-AUTH — 認証機能 設計・タスク計画

関連: DSGN-PROJECT-SPEC
タグ: 認証, セッション, auth, session, task-plan

## TP-AUTH.設計 — 詳細設計
（ユーザーと詰めた詳細設計）

## TP-AUTH.タスク — タスク分解

| # | 識別コード | タスク | 依存 | ステータス |
|---|---|---|---|---|
| 1 | AUTH-SESSION-FLOW | セッション認証フロー | なし | 未着手 |
| 2 | AUTH-TOKEN-SPEC | トークン管理 | AUTH-SESSION-FLOW | 未着手 |
```

**タスクテーブルの依存列:**
- 依存するタスクの識別コードをカンマ区切りで記載する
- 依存なしの場合は「なし」と書く
- state.py の依存チェック機能が /codi 開始時に参照する

**更新モードでの注意:**
- 「完了」のタスクは変更しない
- 「進行中」のタスクを変更する場合はユーザーに確認を取る
- 新しいタスクを追加する場合は既存の番号の後に続ける
- 関連に新しいリサーチページの識別コードを追加する

### 4. インデックス更新

作成・更新後、以下を実行する:

```bash
python "{nxt}/core/index.py" reindex
```

## 重要なルール

1. **識別コードの一意性** — 既存の識別コードと重複しないこと
2. **ページフォーマット準拠** — ヘッダー、タグ行、関連行、セクション記法を守る
3. **170行目安** — 情報量が多い場合はファイルを分割する
4. **関連ページ** — DSGN-PROJECT-SPEC への関連を記載する。リサーチページがある場合はその識別コードも追加する
5. **ステータス管理** — タスクの初期ステータスは「未着手」。`/codi` 実行後に「完了」に更新される
6. **依存列の記載** — タスク間の依存関係を明記する。state.py の依存チェックが参照する
7. **差分チェックの徹底** — 更新モードでは必ず差分チェックを行い、結果をユーザーに報告してから対話に入る
